Mozilla リリースステータス

このページでは、作成中の次期 Mozilla リリースの状態についての情報を提供します。さらに、進捗状況を追跡するために使用する主な Bugzilla クエリーおよび Bonsai クエリーも掲載し、リリースプロセスのどの地点にいるのかを示す手助けとします。

一般的なリリース情報

各 Mozilla リリースは三つの部分に分割されています：アルファサイクル、ベータサイクル、そしてファイナルサイクルです。アルファ開発では、機能開発に集中します。ベータ開発ではクリーンアップをし、全般的なバグ修正に集中します。ファイナルサイクルでは、極めてリスクの低い変更に留め、アルファおよびベータ期間で逃した深刻なバグの修正を行います。

あなたの組織が特定の Mozilla リリースや特定のバグ修正または特定のタイムフレームに依存している場合、drivers@mozilla.org と連絡を取ってください。

リクエストと承認

アルファおよびベータサイクル期間中に問題を突き止め、できるだけ早く修正を施すためにあらゆる努力をしなければなりません。リリースサイクルの終盤にリグレッションや深刻な問題が明らかになった場合、driver はリリースを以前のものよりも優れたものするために必要な変更を特定するために、開発・テストコミュニティに頼ることになります。リリースストッパーとしての認定は、リグレッション、テストおよびレビューがされたパッチの付いたバグ、そして低いリスクで高い効果が得られる時に限る必要があります。

テスターらは、Bugzilla にある blocking? フラッグを利用して、バグをリリースブロッカーとしての認定を行わなければなりません。driver はこれらの認定を評価して、フラッグを + または - に設定します。こうすることで、そのバグがリリースに対してブロッカーであるか、またはリリースに対してブロッカーではないということを示すことになります。

approval? パッチフラッグは、driver による管理期間にレビューされたパッチを該当リリースにチェックインする許可をリクエストするために利用されます。driver はこうしたリクエストを吟味し、フラッグを + または - と設定します。フラッグについてのより詳しい情報 ...

  • リリースブロッカーとなり得るバグ：（blocking1.7? フラッグの付いたオープンバグのための Bugzilla クエリー）
  • チェックインが承認されたパッチ：（ブランチが作成以降、fixed1.7 キーワードあるいは approval1.7+ フラグのついていないオープンなバグのための Bugzilla クエリー）
  • driver の承認を待っているバグ：（approval1.7? フラッグの付いたオープンバグのための Bugzilla クエリー）

その他の注意すべき変更点

私たちは Mozilla ローカライゼーションをビルドしている人たちに作業がしやすくなるように懸命な努力をしています。Beta　のリリースをもって、Mozilla アプリケーションスイート内の全てのローカライゼーションリソースはフリーズされました。稀に Mozilla ローカライゼーションに影響する変更を行う必要が出てくる可能性もあります。このような場合、driver は問題となっているバグに late-l10n というキーワードを付けてフラッグを立てると共に、該当するニュースグループにその変更を伝えます。

  • "fixed1.7" keyword フラグの付いたバグ -- これらの修正をテストして検証するための手助けが必要
  • "verfied1.7" keyword フラグの付いたバグ -- これらの修正は最近のテストビルドで検証済

私たちは Mozilla ローカライゼーションを作成している人たちに作業が楽なるように懸命な努力をしています。ベータ版のリリースを持って、Mozilla Application Suite 内のローカライゼーション文字列は全てフリーズされます。稀に Mozilla ローカライゼーションに影響するような変更を行わなければならないこともあります。そうした事態が発生した場合は、driver らが「late-l10n」というキーワードと共に問題のバグにフラグを立てると共に、適切なニュースグループでアナウンスをする予定です。

  • 締め切り後のローカラズ可能な文字列への変更：（late-l10n キーワードが付いた認証済 1.7 パッチのための Bugzilla クエリー）

特定のリリースサイクル内に Mozilla に対して行われた変更を 100% 正確に反映するような簡単なクエリーを作成することは難しいのですが、大まかな推定はそれほど難しいことではありません。サイクル内でどのような変更が行われたかを見やすくするために、Bonsai と Bguzilla が役立ってくれます。

  • 最近三日間にチェックインされた全ての変更：（最近の 96 時間に行われた変更のための Bonsai クエリー）
  • 1.7 サイクル内で修正されたバグ：（2003年 9月 10日から現在に至る間にされた修正の決定のための Bugzilla クエリー）
  • Mozilla 1.7 に向けたトランクへの全てのチェックイン：（2003年 12月 18日から現在に至る間にトランクへチェックインされた修正への Bonsai クエリー）

フラッグについてより詳しく

先に述べたように、drivers@mozilla.org は二つのタイプのフラッグを利用して、リリース、バグフラッグそしてパッチフラッグを管理しています。

一番目のタイプとして、（バグの Cc: セクションの下に表示される）バグフラッグがあり、現在は blocking1.7[? + -] のように表示されます。全ての Browser および MailNews バグ（NSPR、NSS、PSM および Directory を含む）で、このフラッグを見かけることでしょう。バグを drivers@mozilla.org に推薦する際に利用されます。これまではキーワードや追跡バグを使って同様のことをしていました。

あるバグをリリースのブロッカーとして推薦するには、フラッグを blocking1.7? と設定してください。? とは、1.7 ブロッカーとしてこのバグを評価するように driver にリクエストすることを示します。このバグが原因でリリースを止める場合は、driver がそのバグを blocking1.7+ とします。このバグではリリースを止めない場合は、そのバグを blocking1.7- のようにします。あるバグが - とされたからといって、driver がその修正を必要としていない、あるいは修正に値しないと考えているわけではありません。これは単に driver がその修正のためにリリースを止めることはないということを示しているに過ぎません。

二番目のタイプのフラッグは、パッチフラッグです（添付フラッグとも呼ばれます）。開発ロードマップ の図の中で赤く示されているように、開発フリーズ期間内にある場合、このフラッグは、review[? + -] および superreview[? + -] フラッグの下にあるパッチマネージャの中にあるフラッグリスト内に approval1.7[? + -] として表示されます。認定フラッグは、driver が管理するマイルストーン閉鎖期間内にパッチをチェックインするための認定をリクエストするために使用されます。これまでは、電子メールを使って認定を管理してきましたが、量が増えたために電子メールは実用的ではなくなってしまいました。

driver が管理する 1.7 ツリーの中にチェックインする必要のあるパッチを認定するようにリクエストするには、自分のパッチのところに行き、フラッグを approval1.7? に設定してコメント欄に、パッチをチェックインすることの重要性、その修正に伴うリスク、そして行われたテストについて説明したコメントを入力してください。drivers@mozilla.org がパッチがブランチ・リリースにとって重要だと合意した場合は、フラッグが approval1.7+ に設定されます。これは、パッチをチェックインしても良いという明示的な許可になります。認定をもらったら、最初の ツリー開放 時にチェックインしてください。そのリリースに対してパッチを受け入れたくない場合は、drivers はそのパッチを approval1.7- として設定します。
