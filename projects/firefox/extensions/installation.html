拡張機能のインストール

はじめに

前提条件

このドキュメントを読む前に、拡張機能の構造 と 拡張機能マニフェスト をご覧ください。

このドキュメントは、拡張機能の構造 で説明した、マニフェストファイルだけを使って設計された拡張機能 XPI の自動インストールについて取り上げます。カスタムインストールスクリプトを使った拡張機能のインストール方法については 拡張機能のカスタムインストール をご覧ください。

Web ページからのインストール

ブラウザクライアントは、XPInstall が有効なものとそうでないものという 2 種類に分けられます。XPInstall が有効なクライアントは、ソフトウェアインストールが有効になっている Firefox で、XPInstall が有効でないクライアントは、Internet Explorer など他のブラウザや、ソフトウェアインストールが無効にされている Firefox です。以下、XPInstall が有効なクライアントを XPI+、そうでないクライアントを XPI- と呼びます。

ターゲットアプリケーションにも 2 種類のクラスがあります。ひとつは XPInstall が有効なブラウザ (Firefox)、もうひとつは拡張機能をサポートした他の XUL アプリケーションです。前者の場合、Web ページから XPInstall API を使って直接インストールできます。後者の場合、ホスト環境が提供するコンテンツタイプ処理機能を使ってインストールする必要があります。詳しくは ターゲットアプリケーション をご覧ください。以下、前者の場合を TA+、後者を TA- と呼びます。

2 種類の拡張機能を考えてみます。拡張機能 A は至ってシンプルで、他のアプリケーションには依存せず、1 つの XPI にパッケージされています。拡張機能 B は別の拡張機能 C (バージョン 3.2 から 3.5 まで) に依存します。これはよくある例で、多くの拡張機能は JSLib に依存します。この依存関係の鎖は再帰的な構成が可能で (理論上、拡張機能 C はまた別の拡張機能 D にも依存できます)、複数の依存関係も考えられます。ただし今回は A と B->C の場合だけを想定します。

XPInstall が有効なブラウザクライアントの場合、基本的に、ブラウザから直接依存関係を特定して (下の versionExists 関数を参照) インストールを一括処理できます。そうでない場合は、XPI のリストを用意しておき、ユーザにひとつずつインストールさせることになります。今後、おそらく XRE レジストリを使うことによって、このプロセスを改良できると思います。

  function versionExists(aInstallRegKey, aMinVersion, aMaxVersion)
  {
    var minRV = InstallTrigger.compareVersion(aInstallRegKey, aMinVersion);
    var maxRV = InstallTrigger.compareVersion(aInstallRegKey, aMaxVersion);
    return (minRV >= 0 && maxRV <= 0);
  }

ファイルのインストール

InstallTrigger によって、あるいはホスト環境のコンテンツ処理システムによって、ターゲットアプリケーションの XPInstall システムが呼び出されます。

XPInstall システムは XPI ファイルのトップレベルにある install.js ファイルを探します。これが見つかった場合は 拡張機能のカスタムインストール が呼び出されます。見つからなかった場合は XPI ファイルのトップレベルにある拡張機能マニフェスト RDF ファイルへのストリームを開き、それを拡張機能マネージャに渡します。

拡張機能マネージャはこのストリームを読み込んで RDF データソースに書き込みます。それから以下のアクションを実行します。

  • chrome:targetApplication アークの確認 - これによって、他のアプリケーション用の拡張機能や、必要なバージョンの範囲がターゲットアプリケーションと重ならない拡張機能 (例えば、Firefox 1.0 以上 1.5 未満が必要なのに、インストールされているバージョンが Firefox 0.9 だった場合など) がインストールされるのを防ぐことができます。
  • chrome:requires アークの確認 - 依存関係にある拡張機能がすべてインストールされていなければ、その拡張機能をインストールしません。これによって、スクリプトやバージョンの非互換性などによって引き起こされる問題を防ぐことができます。
  • 以下のアクションはいずれも、拡張機能フォルダにある install.log ファイルにログが記録されます。
  • 拡張機能フォルダの中に、マニフェストに書かれた拡張機能の ID から生成された固有の ID が付けられたフォルダが作成されます。
  • それから、トップレベルのマニフェストを除く XPI アーカイブの内容が、目的の拡張機能フォルダに展開されます。
  • chrome:file アークが列挙され、各パッケージが提供するそれぞれのパッケージ、スキン、ロケールについて registerChrome が呼び出されます。
  • ここまでの処理がすべてうまくいったら、マニフェストデータソースの内容がアプリケーションの拡張機能データソースにマージされます。
  • 与えられた拡張機能の chrome:toBeEnabled アークが true に設定されます。 これで、次回起動時に拡張機能のオーバーレイが有効になり、デフォルト設定が読み込まれます。
  • プロファイルディレクトリに .autoreg ファイルが置かれます。これで、次回起動時にコンポーネントが再登録され、拡張機能コンポーネントがすべて登録されます。

ここで説明した標準的な方法や カスタムインストール でインストールされた拡張機能のアンインストール方法については、拡張機能のアンインストール をご覧ください。
