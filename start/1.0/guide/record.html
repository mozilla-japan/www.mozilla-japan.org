「ブラウザ戦争」は終わっていないのですか？ Mozilla はインターネットに影響するのですか？

地球のデジタル共有地、インターネットは、私たちの生活の中で重要性を増しています。その活力への鍵は相互運用性です - オープン・デベロップメント、オープン・スタンダード、オープンソース・ソフトウェアは、グローバル E-mail アドレッシング・配達システムのような、その他の重要な国際的システムと同じように、第一にインターネットの誕生へ導いたものです。

このオープンの精神を引き継いで、Mozilla 1.0 のコードと開発ツールは、Web 上のコンテンツやデータの表現を自由に制作し、表示する必要のある資源とともに、開発者に用意されています。

Mozilla プロジェクトの目標は、Web 上のコンテンツをオープンにしておくための、スタンダードに準拠したクライアント テクノロジーの創造を革新し、容易にすることです。ますます多くのプログラマーや企業が、戦略的技術として Mozilla を採用するにつれて、Mozilla 1.0 は、インターネットを越えてオープンソースでスタンダードに準拠したソフトウェアの拡散と採択をさらに助長し、今度はその活力に貢献します。

 

なぜ Mozilla 1.0 は完成までこんなに時間が掛かったのですか？

まず、プロジェクトの領域が劇的に変化しました。当初のプロジェクトは、Netscape Communicator 4.x 製品ラインから、計画された 5.x 製品ラインへの、増分のみのアップグレードでした。しかし、そのプロジェクトが立ち上がってから、その目標の多くが、現存のコードベースを使って取り組めないということが明らかになりました。

そして、決定は Mozilla プロジェクトをリセットし、新しいコードの形を生成させるということになりました。この決定がプロジェクトの領域を劇的に拡大したのです。

次に、新しいコードベースに向けて、すべてのデザイン変更が一度に行われませんでした。開発プロセスは家を改築するような大変なことだということが分かりました。理想的には、1 つの計画は始める前に詳細が正確で、完璧ではなくてはなりません ・・・ しかし現実には、これはめったに可能なことではありません。

最後に、これは私たちの 1.0 リリースではありますが、世間一般に製品の最初のリリースに適合したスタンダードによって評価されることはないでしょう。代わりに、Mozilla 1.0 は最新の商用ブラウザの世代と対照して比較されるでしょう。そのため私たちは、このリリースが本当に「プライム・タイムの準備ができた」かどうかを確かめる必要のために時間を費やしました。

 

ユーザ体験が本当に 革新を得るのはいつですか？

機能性、そしてユーザインターフェースの両方の観点から、Mozilla 1.0 の強調すべき点は、Netscape が信頼できる Netscape Communicator 4.x シリーズに取って代わるものを創造させたことでした。しかしながら、ユーザ体験の将来の発展と特に関係があり、Mozilla がエキサイティングで速やかに発達する未来を自称する覆いの下には、重大な変化がありました :

  • 機能性は、4.x シリーズの中にあったものよりずっとモジュールで拡張性があります。
  • 全体のユーザインターフェースは XUL を使って定義されていて、これはだれもが Mozilla のユーザインターフェースによって実験できる、スタンダードの Web ページデザイン技術を使うことができるということを意味しています。

たとえば、タブブラウジング機能を考えてください。H. J. van Rantwijk はこの機能を他のブラウザで使いました。昨年、彼と何人かの友人は Multizilla を制作するために XUL を使いました。David Hyatt はその作品を見て、すぐに基本的な特徴を最初から実装した、新しい XUL 部品を作成しました。基本的なタブブラウジング機能は現在標準の Mozilla の機能セットの一部となっていて、さらに洗練された Multizilla プロジェクトの作業は続いています。

Mozilla プロジェクトとそのまわりの開発コミュニティは、常に上達するプロジェクトを創造するために協力して作業しています。そして私たちは、ユーザインターフェースの実験と製品の実装は、一度 Mozilla 1.0 を世に送り出せば加速するだろうと予想しています。

 

「バグ・カウント」は上がり続けていると聞きました。これは本当ですか？

「バグ・レポート」は、多様な人々を巻き込んで公然と追跡することが十分に重要な、Mozilla プロジェクトに関係するどんな問題についても、Bugzilla で作られることができます。私たちのバグ報告・追跡システムは、Mozilla プロジェクトの開始から作られたすべてのレポートを含んでいます。そしてデータベースには解決したバグが残っていますので、この数は必然的に増えるのです。

プロジェクトの全般的な品質のバグ・レポートの数について話す正確な方法はありません :

  1. 一部のレポートはプロジェクトの活動に関するもので、コードに関するものではありません。
  2. コードに関係した重大なバグの数は、その他のバグとまったく同じものか、悪性のバグ・レポートのどちらかで、一部は特定の機能強化についての意見、あるいはリクエストの表現です。
  3. 新しい特徴へのリクエスト（RFE、「機能強化へのリクエスト」）はバグ・レポートとして追跡されます。
  4. 私たちは「メタ・バグ」のシリーズを持っています。これはプロジェクト管理のために使われます。たとえば、そのリリースについて重要な問題点を追跡する、各マイルストーン リリースについてのメタ・バグがあります。
  5. 整理されているバグ・レポートの数は、プロジェクトの関係者の数に影響されます。多くの関係者がいれば、それだけ Bugzilla へたどり着くバグ・レポートが多くなります。
  6. バグ・レポートの存在は、少しでもユーザへのバグの影響について考えれば小さいものだと言えます。1 つのバグ・レポートがすべてのユーザに矛盾なく影響する巨大な問題を含んでいるかもしれませんし、ほとんどのユーザがめったに遭遇しない、大変特殊な問題を含んでいるかもしれません。バグ・レポートがますます多くの珍しい問題を扱うにつれて、レポートの数が、全般的な品質の向上について同時に増えるのは、実際あり得ることなのです。

バグ・レポートの数は「bugginess」の追跡を手助けしませんが、私たちは他の bugginess、特に標準的な製造概念である平均故障間隔 (MTBF) の大きさを追跡します（たとえば、近頃のビルドの この分析 をご覧ください）。

 

mozilla.org と Netscape や AOL のようなスポンサー企業との関係は何ですか？

Mozilla.org は Mozilla を成功したオープンソースプロジェクトにすることを見いだします。これはすべての Mozilla コミュニティをサポートし、Mozilla コードベースを利用したり改良することに興味がある人たちのために、連絡とコミュニティの主要な場所を提供します。

Mozilla.org は、意志決定において、完全な独立を維持します。決定は、コードベースとすべての Mozilla コミュニティの、長期の健全な状態のために役立てられます。Mozilla.org は、スタッフの場所や資源（サーバその他）について企業の貢献を受けています。また、コミュニティから入手できる最高の寄付を受けていますし、販売業者とも緊密に連絡を取っています。

Chief Lizard Wrangler を含む、mozilla.org の一部のメンバーはボランティアです。他のスタッフメンバーは雇い主によって後援されています。すべての場合において、むしろプロジェクトから起こった特定の製品よりも、プロジェクトそのものへ献身が必要とされます。Mozilla.org には、完全な透明性への長年の責任があります。

 

Netscape がほとんどの作業（コードやドキュメントの記述、テストなど）を行っているのではないのですか？

Netscape は Mozilla プロジェクトに対する最大単独の貢献者で、たくさんの前述のボランティアを雇い、彼らの多くをプロジェクトにフルタイムで働かせることによって、コミュニティを元気づけてきました。

Netscape の従業員は、数少ない主要なボランティアや、増え続けている、Mozilla のコードを使っているその他の企業の従業員と常に協力して、コードベースの大部分の側面に親密に関係してきました。今日まで、Netscape と関わりのない、およそ 300 人の人々が、プロジェクトに対してコードの貢献者になってきました。

プロジェクトの他の側面に参加している人々の数は、それよりも一層多くなります。特に、製品テストやバグ報告に関わっている人たちの数は莫大です。近頃の mozilla.org のリリースは毎回 30 〜 40 万の人々にダウンロードされていて、プロジェクトに優秀でタイムリーなテストデータを与えてくれています。

また、コードを書いているそれらの開発者を手伝うテストの活動に参加している人たちは何百人もいます。これは開発者がさらに有能で精鋭であるようにしています。

同様に、様々なチュートリアルを含め、生成されているドキュメンテーションの多くはボランティアのコミュニティによるものです。

 

AOL は Mozilla を利用するのですか？

Mozilla.org は、将来の AOL の行動について話すことはできません。それができるのは AOL だけです。AOL の 100 パーセント子会社である CompuServe は、現在 CompuServe 7.0 クライアントの一部として、「Gecko」として知られている Mozilla テクノロジーを出荷しています。AOL は現在こちらも Gecko を使った AOL 7.0 クライアントのベータ版をテストしていますが、どんな将来の計画についてコメントすることも時期尚早です。

 

Mozilla コードベースを管理しているライセンス条件とは何ですか？

Mozilla コードベースはオープンソース／フリーソフトとして開発、認可されています。これは、ソースコードの受け取り側が基本的な権利を保証されているということを意味します。

Mozilla コードベースの大部分は Mozilla Public License と Netscape Public License によって管理されています。互換性のあるオープンソース／フリーソフトライセンス（BSD や MIT ライセンスのような）のファイルも含まれるでしょう。

これは Mozilla コードベースが合法的であり得るコードベースであり得るということを意味しています :

  • 表示 （http://lxr.mozilla.org/）
  • 変更 （http://mozilla.org/cvs.html）
  • 使用 （http://mozilla.org/build/）
  • 配布 （http://mozilla.org/MPL/）

は、目的を問わず、どんな個人、企業によっても行われます。

オープンソースとフリーソフトについての詳しい情報は、http://opensource.org や http://www.fsf.org でそれぞれご覧いただけます。

コードベースは現在、同じフリーでも 互換性のないライセンス （GNU GPL と LGPL） を持つ他のプロジェクトのコードと Mozilla のコードを、簡単に、合法的に化合できるようにするため、リライセンスされています。遅延は 「行方不明」の貢献者たち のひと握りです。

 

他にどんな技術がありますか？ それを便利だと思うのは誰ですか？

Mozilla のコードベースは幅広い種類の、比較的モジュラー式でスタンダードベースの機能性を含んでいます。開発者、特に OEM は、そのコードベースが、移植性がありスタンダードに準拠した Web やネット関連ソフトの作成、拡張のための、魅力あるフリーの「プラグ・アンド・プレイ・グラウンド」な品質の高いソフトのソースであることに気付くでしょう。

その機能性の多くは Mozilla 1.0 で使われていますが、一部は使われていません。

たとえば、Bugzilla のような開発者ツールがあります。これは Mozilla プロジェクトの範囲から離れた外側で成功している、Mozilla テクノロジーの見本です。Mozilla とは関係のない他のオープンで私営のプロジェクトで、現在、情報の追跡のために世界的に使われている数千の Bugzilla インストレーションがあります。

さらに、Java による JavaScript の実装である Rhino のような、様々なその他のプロジェクトもあります（これは Java ソリューションを開発している人が、エンドユーザに対してスクリプト能力を提供できるようにします）。

 

mozilla.org では何が起こっているのですか？

それが何であっても、それは著しくオープンです。最初の「ロードマップ」ドキュメント（1998 年 10 月）にある最初の設計原則がまさにその問題について触れていることは、注目に値することです（他の原則は技術的な問題に関するものでした）。

Web サイトはふつう、開発者によって開発者向けに書かれています。それはたいてい、報道関係者やエンドユーザのようなその他の読者たちを無視し、また彼らによって無視されます。もしあなたが十分な技術的、実際的知識を持っていれば、Web サイト編集・アーカイブ機能や、様々なプラットフォーム上で動く様々な製品ビルドのリアルタイム・ステータスの、ごまかしの見た目が単純でグラフィカルな表示のような、印象的なツールが手に入ります（例 : ある Mozilla プロジェクト ブランチのための最新のステータスの この表示）。

Mozilla 開発は、定期的に更新される「ロードマップ」ドキュメントで定義されている方針をたどります。優先順位と目標は mozilla 1.0 宣言 のような「宣言」ドキュメントで定義されています。
