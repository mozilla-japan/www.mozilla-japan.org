Rhino 1.5 の最新情報

ECMA 262 Edition 3 の適合

Rhino 1.5 は ECMA 262 Edition 3（時々「 ECMAScript 」と照会される）に適合した JavaScript 1.5 を実行します。Edition 3 は、下に記載されている JavaScript 1.4 ですでに与えられていた JavaScript の特徴を標準化しています。
  • 正規表現
  • switch 文
  • do...while ループ
  • 文のラベルとラベル付き break と continue 表記ラベルと break と continue ラベル
  • オブジェクト・リテラル
  • ネスト化された関数
  • 例外ハンドル
  • instanceof 演算子
  • in オペレター
その他に、Edition 3 と JavaScript 1.5 に追加された新しい機能は下記の通りです。
  • 貪欲な量指定子の様な演算子を含む Perl 5 の正規表現
  • 例外として捕捉されたエラー
  • 数字整形 (Number.prototype.toFixed, Number.prototype.toExponential と Number.prototype.toGeneral)

Rhino 1.4 Release 3 からの変更

最初のリリースからオープン・ソース (1.4 Release 3) になるまでの、その他の重要な変更点は下に記載します。ここではバグ修正については述べられてなく、API の変更と重要な機能変更のみです。

コンパイル・モード

Rhino は実行可能な二つのモードを持っています。インタープリタ・モードは Java によって実行されるインタープリタ・ループを持っており、コンパイル・モードは JavaScript コードをクラス・ファイル中の Java バイト・コードにコンパイルします。このコンパイル作業は、すでにインタープリタ用に利用可能な API を使ったスクリプト・エバリュエーションの一部として、もしくは、別個のコンパイル・タイム・ステップ中に行われます。そのインタープリタ用のコードは org.mozilla.javascript.optimizer のパッケージの中にあります。
 

JavaScript コンパイラ

現在、このディストリビューションはコマンド・ラインから呼ぶことのできる特別はクラスを含んでいます。これは jsc, the JavaScript compiler です。このツールは JavaScript から Java クラスを生成する為に使います。 JavaScript のスクリプトが applet や servlet のような重要なプロトコルを実装できるように、自由にインターフェースを実装したり、自由に基底クラス拡張した Java のクラスを作成できるようにするオプションがあります。 http://www.mozilla.org/rhino/jsc.html を参照してください。
 

LiveConnect 3

現在、Rhino は LiveConnect 3 の仕様 (LC3) をサポートしています。一番の著しい変更は overloaded method resolution をサポートした事です。 LiveConnect Release 3 Goals/Features を参照してください。
 

Java のプロパティを反映した JavaBeans のプロパティ

getFoo/setFoo メソッドを持つ Java のクラスは JavaScript の反映中に「 foo 」というプロパティを持ちます。ブーリアン・メソッドも、また、反映されています。
 

動的なスコープのサポート

Rhino 1.5 の実行は、特に、サーバーの組み込みの様な、マルチスレッド環境で有効な、動的なスコープをサポートしています。
 

ScriptableObject.defineClassの為の新セマンティックス（意味論）

JavaScript オブジェクトを定義する為の古いルールは奇異になってきました。現在、それらのルールは、まだサポートされていますが、より偽りのない定義がサポートされています。詳しくは javadoc 参照してください。
 

Java 2 の -jar オプションをサポート

現在、Java 2 の新しい -jar オプションを使って、シェルをスタートさせる事が可能になりました。
 

シェルの変更

ここでは二つの変更点について紹介します。最上位変数での「 environment 」と「 history 」の追加です。
 

スクリプトに明白な Java クラス

ある JavaOne の出席者が、多くの組み込みが全ての Java クラスにアクセス可能なスクリプトを必要としないのではないか、という指摘しました。これは素晴らしい指摘で、私は、どのクラスがスクリプトにあらわにされるか組み込みに選択させる事を許す SecuritySupport というインターフェイスを追加しました。
 

SecuritySupport と JavaAdapter

Andrew Wason さんは新しい JavaAdapter の特徴（クラス・ファイルの生成によって、JavaScript オブジェクトに、任意の Java インターフェイスを実行させる事を許す）の問題を指摘しました。それは、Rhino にバイト・アレイから組み込みによって供給されたルーチンへのクラスの生成を委譲させる SecuritySupport インターフェイスをサポートしていませんでした。この能力は、クラスの生成は特別な動作と考える為、セキュリティの見地から重要です。

私は、この問題を修正する為の変更を mozilla.org の CVS リポジトリに送りました。もし、Context が生成された時、SecuritySupport クラスが指定されたら、JavaAdapter の使用が、クラス生成を SecuritySupport クラスに委譲します。
 

Context.exit()

Context.exit() はインスタンス・メソッドから静的メソッドに変更されました。これはまた、それ（Context.exit()）に、静的な Context.enter() メソッドを適合させます。
 

Context.enter(Context)

新しい Context.enter からのオーバー・ロードが追加されました。この追加作業無しでは、存在する Context をスレッドに加える事は可能ではありませんでした。詳しくは javadoc を参考にしてください。
 

Listeners for Context

Context は現在、その二、三の属性として、属性変更 listeners をサポートしました。
