Mozilla Thunderbird の将来

Thunderbird ユーザのみんな、あけましておめでとう! 新年の抱負として、2004年の Thunderbird に関して考えている計画とアイデアについて述べたい。この文書は新しい Thunderbird ロードマップだろうか? そうかもしれない。しかし、ロードマップは特定のマイルストンとスケジュールについて述べるものだ。今回はそこから離れて、今年フォーカスすべき領域とトレンドに注目したい。これは詳細なリスだろうか？ いやまったく違う。このリストはみんなのお気に入りの機能とウィッシュリストを含んでいるだろうか？ いや、この小さなドキュメントでそれらすべての要求に適合させることができるとは思わない。そしてもちろん、私たちが反映したいと思っている電子メール空間にどんな面白い開発が起こるか誰が知っているだろうか。

混乱を避けるために、特別な順番を持たないリストを掲載する。

Thunderbird インストーラ

私たちは、Mozilla Thunderbird のインストーラを開発するために、Ben Goodger の Mozilla Firebird のインストーラ作業に長い間投資してきた。インストーラを持つことは新規ユーザが Thunderbird を導入したりちょっと使ってみたりするのを容易にするのに大きく影響する。加えて、基本となるインストーラを持つことにより、共通の拡張（とくにオフラインサポート）のオプションをインストーラーの UI に組み込みたいと思っている。これによりこれらの拡張と一緒のアップグレードがより堅牢になる。

Seamonkey プロファイルの取り込み

現在、ユーザは Mozilla Application Suite からプロファイルを手動で取り込むことができる。これは、Mozilla の pref.js で相対パスを使うという最近の修正によりより簡単になった。しかし、私たちは自動取り込みの手段を必要としている。優れた取り込みのためのソリューションは二つの動作を持つだろう: (1) 既存の seamonkey のプロファイルをそのまま再利用する機能と、 (2) すべてのメールフォルダとデータを Thunderbird のプロファイルにコピーする完全取り込み機能だ。前者は、ユーザが Seamonkey と Thunderbird の両方で重複したメールデータをローカルディスクに作ることなくメールを読むのを容易にする。後者はユーザが suite から Thunderbird に移行しやすくする。

ジャンクメール改善

私たちは Mozilla Mail プロダクトのベイジアンベースのジャンクメール検出の開発に大きな実績を残した。しかし、スパマーはこれらのコントロールを回避するために異なる手段を使って送り続けている。私たちもこの世界で前進し続けていく必要がある。取り組もうとしているアイデアがいくつかある:

  • メッセージがジャンクであるかを決定する分類アルゴリズムの改善
  • サーバーサイドジャンクコントロールのサポート。多くのサーバはジャンクと思える受信メールにヘッダを追加する。これらのヘッダを検出し、メッセージをジャンクとマークし、ジャンクと認識されたメッセージを処理するためにユーザが定義した動作に従う機能が必要だ。
  • ベイジアン分類に送られる前に、メッセージを単語分割するためのルールの改善。これはよい結果が得られるはずだ。
  • トレーニング集合に保存するトークン数の制限。これはジャンクコントロールのメモリフットプリントを削減し、さらに性能を少し改善させるだろう。

スペルチェッカ

我々のスペルチェッカは、Open Office で使用される myspell エンジンの2年前の分岐をベースにしている。分岐してから、多くの改善が Open Office バージョンになされ、ウェブ上で最も優れたスペルチェッカの一つになった。もし元に戻って、それらを我々の分岐に取り込むことにより最新の Open Office に見られるのと同じ水準にもって行くことができればすばらしいことだ。

アカウントごとに複数の ID のサポート

われわれの多くは同じドメイン名の複数の電子メールアカウントをもっている。つい最近までは、一つの電子メールアカウントについて一つの ID しか持つことが出来なかった。ずるいユーザは今では prefs.js を手で編集して 一つのアカウントに複数の ID を関連付けすることができる。メッセージに返信するときに、元の送信者があなた宛に送信するときに使用した ID を compose ウィンドウでデフォルトの ID として選択するとスマートだろう。しかしながら、我々はアカウントごとに複数の ID を作成したり管理したりするきれいな UI をまだ持っていない。これの難しい点は、きれいな UI をデザインすることだろう。

プロファイルのポータビリティ / USB デバイス

これは Thunderbird が便利になるだけでなく、Mozilla のプロファイル全般にかかわる問題だ。アプリケーションのプロファイルを USB キーチェインドライブで運ぶことにより、マシン間でプロファイルのポータビリティを実現したいと考えているユーザが増えてきている。それには解決すべき二つのクラスの問題がある: (1) ドライブレターが変化するリモードドライブの上のプロファイルで動作するようなインストール、(2) アプリケーションとプロファイルが両方ともリモードデバイスにあるようなケース。

Talkback 統合

クラッシュレポートデータを生成するために、Talkback テクノロジーを Mozilla Thunderbird に統合する。もちろん、クラッシュしないことに越したことはない！

複数の SMTP サーバーのオーバーホール

現在複数の SMPT サーバーをサポートしている。しかし、SMPT サーバーをアカウントに関連付ける方法と、ユーザがどのようにして利用する送信 SMTP サーバーを選択するかの方法はごちゃごちゃだ。複数の SMPT サーバをユーザにどう見せてそれをどう管理するか問題を見極める必要がある。

RSS フィード統合

RSS フィードをメールウィンドウに統合させる。メールは RSS フィードを実現する自然なアプリケーションだ。ぜひそれを利用しよう。

メッセージリストペインにグルーピングを追加

メッセージの"履歴のような"ビューを実現するのはどうだろうか。例えば、日付でソートすると、"今日のメッセージ"、"昨日のメッセージ" などが得られる。送信者でソートすると、"Scott からのメール" だ。

添付ファイル

メッセージから添付ファイルを削除する機能。コンテンツタイプやファイル名をベースに添付ファイルを検索する機能。
