テキスト・ウィジェットにおけるエディタ

  • エディタのインスタンス化
  • イベント・ハンドリング

はじめに

Mozilla のエディタ は、XUL や HTML テキスト・ウィジェットにも使われています。こうすることによって、テキスト編集用のコードベースを一つにすることができます。コードベースは国際的なテキスト入力 (IME) を扱っていて、CSS を用いてスタイルを追求することができるようになっています。しかしながら、テキスト・ウィジェットにおいてエディタに必要なことはエディタを composer で利用する場合と幾分異なっているので、エディタはいずれの状況にも応じられるように柔軟性を持つ必要があります。主な違いは次の通りです：

Composer テキスト・ウィジェット
エディタが動作するのは： ドキュメント全体 ドキュメントのサブツリー
機能の範囲： リッチ HTML の編集 プレーンテキストのみ
ホスティング・コード： エディタ・シェル レイアウト・フレーム 及び コンテント・コード

テキスト・ウィジェットでは、エディタはフレーム ( 言い換えるとレイアウト・フレーム・オブジェクト ) によってテキスト・ウィジェットのために作成され、ホストされます。 nsGfxTextControlFrame2 というフレームです。このフレームはエディタのためのキー押下とフォーカス・イベントをフィードバックするイベント・ハンドラを作成して登録し、それからエディタ API を呼び出すことによってテキスト・ウィジェットの値を取得して設定します。

テキスト・ウィジェットとしてエディタを最初に実装したときには、各テキスト・ウィジェットがそれぞれ独自のサブドキュメントと関連したデータ構造を備えていて、エディタはそのサブドキュメント ( これが元々の nsGfxTextControlFrame です ) 上で動作するという、重量級の試みを用いました。最近では、(一般的に ender lite と言われる) ある方法を採用しています。その中ではテキスト・ウィジェットのコンテントが実際には親ドキュメントの中にありながら、匿名のコンテントに なっています。匿名のコンテントは標準的な DOM API ではアクセスできませんが、特別な使い方ならアクセスできるドキュメント・コンテントです。ender-lite を使えば、 エディタは親ドキュメント ( HTML ドキュメントでフォーム・コントロール・ウィジェットのために使うもの、または XUL テキスト入力フィールドのために使われる XUL ドキュメント ) のサブツリー上で動作します。この場合には、エディタは、関与してもよいコンテント・サブツリーのルートを「知って」います。

エディタのインスタンス化

テキスト・ウィジェットのためのフレーム構築は、HTML の中に <input type="text"> を、または XUL の中に <textfield> を見出してレイアウトがフレームを構築するときに起こります（ これが最初のページ読み込みの際に起こること、そしてスタイルシートの変更などのようにフレームを再構築させるような他のどのようなことの後でも起こることに注意してください ）。nsCSSFrameConstructor::ConstructTextControlFrame() がテキスト・コントロール・フレームを作成し、その後すぐに nsGfxTextControlFrame2::ÈB;CreateAnonymousContent() が呼び出されます。( nsGfxTextControlFrame2 が nsIAnonymousContentCreator を実装するからです )。この呼び出しはテキスト・フィールド・コンテントを保持する匿名のコンテントを構築するところにあって、エディタをその上に作成することになります。<input> 要素の匿名の子としてコンテント・ツリーの間に挿入される <div> 要素を作成します。
