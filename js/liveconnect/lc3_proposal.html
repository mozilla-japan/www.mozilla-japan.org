LiveConnect Version 3 目標

課題/特徴
(優先される順番の
降順で記載)
詳細
Thread-safe 性
 LiveConnect の LC2 リリースは完全には thread-safe でなかった。 (OJI のような LiveConnect の元のアプリケーションが厳密な thread-safe 性を 要求しなかったので、LC2 の最優先課題ではなかった。)
Rhino 互換性
 Rhino (Java での JavaScript) と LC2 の振る舞いは、特に例外処理の領域で、 それとなく異なっていた。Rhino と LC2 の両方が 一つのテスト一式が LiveConnect の両バージョンで使えるように 修正されるべきだ。
OJI LiveConnect との一致
 OJI プロジェクトはその要求に適合するように LiveConnect API を変更し、 その結果を別のディレクトリに置いた。 ソースを同じコードベースに統合するするようがある。
予想可能なオーバーロードされた
メソッドの解決
 2 個以上のオーバーロードされた Java のメソッドが JS からの呼び出しで引数の数と型に 互換性がある時、Java のメソッドの選択が、Java のリフレクション API に最初に列挙された ものを使う事によって、任意に行われている。 残念ながら、列挙時のメソッドの順序はどんな仕様にも規定されていないので、 JVM ベンダー間の違いによって、

代わりに、オーバーロードされた Java のメソッドの中から選ぶ JVM 独立な規則一式 を定義した。

明示的なオーバーロードされた
メソッドの解決
 JS 言語の型付けシステムと Java のそれとの間の弱い対応は、 曖昧かつ/もしくははっきりしないという結果になる。 例えば、JS の数値型はあらゆる Java の浮動小数点あるいは整数型: byte, char, short, int, long, float, double にマップできる。

メソッドオーバーロード解決処理を回避し、明示的に 呼び出すメソッドを指定する方法が必要である。例えば:

myPrintMethod = java.io.PrintStream["print(double)"];
myPrintMethod(13);

Java のインスタンスを使った Java の static メソッドの呼び出し
 Java 言語では static メソッドをクラス名もしくはクラスのインスタンスへの参照を 使って呼び出す事ができるが、LiveConnect の現在のバージョンでは前者のみしか できない。
java.lang.String オブジェクトが
JS の文字列メソッドを"継承"する
 LiveConnect の現在のバージョンでは、Java の String を JS の文字列メソッドのレシーバとして 使う前に JS の文字列に変更する必要がある。 一般的には、例えば空文字列を Java の文字列に追加する事で。

    s = new java.lang.String("foo")
    s = s + "";  // s は今 JS の文字列である。
    s.match("oo")

目標は、java.lang.String オブジェクトを、 JS の文字列の全メソッドを"継承"する特別な物と扱う事で、 明示的に JS の文字列へ変換する必要をなくす事だ。 だから例えば上の例の 2 番目の文は不必要になる。

JavaArray オブジェクトが
JS の配列メソッドを"継承"する
 JS の Array の関数は意図的に一般的であるので、reverse() と join() のような JS の配列のユーティリティメソッドを JavaArray オブジェクトに適用 できるべきである。
新しい JS 言語の特徴を
サポート
 ECMA 言語へ追加された instanceof と in のサポートを 追加。
JS の例外に変換された Java の例外。例えば Java を呼び出す JS を Java が呼び出した時の、 Java の呼び出し側に伝播した未捕捉の例外。
