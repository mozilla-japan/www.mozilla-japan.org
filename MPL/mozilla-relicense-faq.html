Mozilla Relicensing FAQ

Version 1.0

先に mozilla.org は、 Mozilla Public License（MPL）および Netscape Public License（NPL）と GNU General Public License（GPL）との間の齟齬を解消するため、新しいライセンスの許で Mozilla コードにライセンスを再付与するという方針を表明しました。この方針に対して、特に開発者が自身のコードに GNU Lesser General Public License（LGPL）（英文）を適用することについて、多くの方々から貴重なご意見を頂きました。

こうした意見や Mozilla への貢献者の間での議論を基に、かつて NPL の許でライセンス付与されていたコードで、現在 NPL/GPL デュアルライセンスの許でライセンス付与されているコードに、NPL/GPL/LGPL "トリプルライセンス" を適用して、第一次の Mozilla 再ライセンスを開始しました。再ライセンスの次の段階では、NPL/GPL デュアルライセンスが適用されている残りのソースファイルに NPL/GPL/LGPL トリプルライセンスを付与し、MPL および MPL/GPL 下のソースファイルに MPL/GPL/LGPL トリプルライセンスを適用します。この FAQ では、こうした再ライセンスに関する様々な質問についてお答えします。その他の新しい Mozilla ライセンス方針については、新しい mozilla.org ライセンスポリシー をご覧下さい。

重要：このドキュメントはあくまで参考で、法的な効力をもつアドバイスではありません。Mozilla コードを基にソフトウェアを開発・配布したい場合や、Mozilla コードを利用する上での権利や義務について法的なアドバイスが必要な場合は、ソフトウェアライセンスについて専門的知識をもつ弁護士にご相談ください。

再ライセンスの詳細

なぜ Mozilla コードは再ライセンスされるのですか？

以前問題とされていたのは、Mozilla ライセンスの方針が GPL および LGPL と "互換性" があるかどうかでした。

NPL および MPL が作成されてから、Free Software Foundation（英文）は、NPL および MPL は GPL とは "互換性がない" と判断しました。ここでいう "互換性がない" とは、（FSF によると）開発者が NPL (または MPL) 下のコードを GPL 下のコードと組み合わせてその成果を配布する場合、GPL で定められた条件に抵触してしまう、ということです。LGPL が GPL と同様の条件を課していることから、実際に MPL が GPL と互換性がない場合、LGPL にも互換性がないことになります。

（LGPL は、LGPL の許でリリースされたソースと、非 LGPL のコードをある特定の方法で組み合わせることを可能にしています。例えば、LGPL が適用されたコードが適用されたライブラリコードを利用してアプリケーションをリリースする場合です。関連する LGPL の条項は、MPL 下のコードと LGPL 下のコードを組み合わせて、より規模の大きな製品を作成する際の、全ての可能性に言及しているわけではありません。特に、Mozilla ソースファイルを直接 LGPL 下のライブラリと組み合わせる（例：ライブラリ機能の拡張）場合、LGPL の効力、および潜在的なライセンス非互換性は、GPL に関する条項と同様のもののようです。）

開発者が、このようなライセンスの非互換性に基づいて著作権の侵害を訴えることができるかは不明です。しかし、この問題に関して不明瞭な点を解消し、開発者が Mozilla コードを、通常 GPL または LGPL の適用を受けるアプリケーションで利用したい場合に問題となる懸念に備えて、Mozilla コードを再ライセンスして GPL および LGPL との非互換性についての問題を解決したいと考えたのです。

私たちはまた、Mozilla ソースツリーにチェックインされた新しいソースファイルに、NPL（NPL 単体、または NPL を基本にしたデュアルまたはトリプルライセンス）が適用されるべきではないという mozilla.org の方針を繰り返し強調しています。その替わり、mozilla.org から特別な指示がない限り、下に示される、新しい MPL/GPL/LGPL トリプルライセンスが、ツリーにチェックインされた全ての新しいファイルに適用されるべきです。この方針は、全ての Mozilla 貢献者に適用されます。詳細情報は、mozilla.org ライセンスポリシーをご覧下さい。

NPL の目的は、Netscape のプロプラエタリーコードベースからオープンソースコードベースへの移行を支援することにあります。この期間に、NPL を使用することで、Netscape は、第三者とのプロプラエタリーソースコードライセンスの規約を遵守することができ、NSPR などの Mozilla 関連のモジュールを Netscape のプロプラエタリーサーバー製品に使用することができるようになるのです。NPL は、既に NPL や NPL を基礎にしたデュアルまたはトリプルライセンス下にある、既存のファイルに適用され続けます。しかし、新しいファイルには適用されません。

（これに関連して、新しい Mozilla ソースファイルを作成している多くの開発者が、使い慣れた別のファイルからライセンスヘッダーを、単にコピーアンドペーストしてしまうことが明らかになりました。しかし、そうすることで、時として開発者は、新しいファイルを誤ったライセンスの許で不適切なライセンスを与えてしまうのです。あなたが新しい Mozilla ソースファイルを mozilla.org がホストするリポジトリーに追加する場合は、新しいライセンス条項に従って、適切なライセンス告知がされているかを、必ず確認して下さい。）

Mozilla 再ライセンスの新しい基本方針は何ですか？

新しい標準的なライセンス方式は、MPL/GPL/LGPL トリプルライセンスです。このライセンスは、mozilla.org を通じて頒布される新しい Mozilla ソースファイルに使用され、必要な許可を得た場合は既存のファイルにも使用されます。（同様の NPL/GPL/LGPL トリプルライセンスは、NPL または NPL/GPL デュアルライセンスの許にある、既存のソースファイルに使用されます。）

MPL/GPL/LGPL トリプルライセンスは、NSPR、NSS、PSM、そして JavaScript エンジンを含む、既に Mozilla コードに使用されているデュアルライセンス方式の拡張版であり、変形版です。一番の違いは、トリプルライセンス方式の代替ライセンスとして、GPL に加えて、LGPL を使用していることです。また、新しいソースファイルには NPL を使用することを奨励していません。

どの Mozilla コードが再ライセンスされるのですか？

私たちは、mozilla.org を通じて頒布される Mozilla ソースコードディストリビューションに含まれ、また、mozilla.org を通じて頒布される Mozilla バイナリーディストリビューションを作成するのに使用される、全てのソースコードに再ライセンスを付与することを目指しています。私たちは、必ずしも、mozilla.org の CVS リポジトリーから利用できる全てのソースコードを再ライセンスしようとしているのではありません。

私たちの第一の目標は、Mozilla バイナリーディストリビューションをビルドするのに必要なソースコードが、適切なライセンス（または複数のライセンス）の許に置くことです。そのため、私たちは、mozilla.org を通じてリリースされる、Mozilla ソースコードディストリビューションに含まれるソースコードの再ライセンスに焦点を合わせています。（このようなディストリビューションの例は、0.9.3 マイルストーン用のソース tarball があります。）

すでに再ライセンスされている Mozilla コードには何がありますか？

かつて NPL の許でリリースされた Mozilla ソースファイル（NPL/GPL デュアルライセンス下でリリースされた他のソースファイルと並んで）です。こうしたソースファイルには、Netscape が他者によって頒布されたコードに再ライセンスする権利があります。

AOL Time Warner 管理部門は、Mozilla ライセンス方式を見直して、Netscape から著作権を付与された Mozilla ソースコードと GPL/LGPL との非互換性の問題について対処することを認めました。これにより、NPL の NPL セクション V.3（"代替ライセンス"）下でライセンスを付与されているソースファイルについては、Netscape そして、現在の AOL Time Warner がそうしたファイルに再ライセンスができるようになりました。こうしたファイルの中には、他者や、その他の（NPL 条項以外の）ライセンス条項の許で、第三者によって提供されたあらゆるコードが含まれます。

私たちは、現在 MPL でライセンスされている Mozilla ソースファイルのライセンスを変更するために、この条項を利用しました。つまり、そのようなファイルのための新しいライセンスは、前述した MPL/GPL/LGPL に似た、NPL/GPL/LGPL トリプルライセンスなのです。私たちはまた、現時点で NPL/GPL の許でライセンスされている Mozilla ソースファイルのいくつか（例えば、js ディレクトリ内の JavaScript エンジンソースコード）のライセンスを変更するために、この条項を利用しました。繰り返しますが、新しいライセンスは、NPL/GPL/LGPL トリプルライセンスでもあるのです。（現在 NPL/GPL デュアルライセンス下のソースコードを、全て再ライセンスしたわけではありません。なぜなら、私たちは、そのために必要な許可を全て取得したわけではないからです。）

まだ再ライセンスされていない Mozilla コードには何がありますか？

現在、MPL や MPL/GPL デュアルライセンス下のソースファイルです。こうしたライセンスでは、残りのソースファイルが MPL/GPL デュアルライセンスのままであるのと同様に、非 Netscape 配布者からコードを再ライセンスする追加の許可が必要となります。

前述の質問にもあるように、AOL Time Warner は、Netscape による著作権コードについて、GPL/LGPL の非互換性問題に取り組むために、Mozilla ライセンス方式を変更する承認をしました。つまり、現在 MPL または MPL/GPL デュアルライセンスの許にあるソースファイルは、もしファイル内の全てのコードが、Netscape によって著作権が与えられている場合（その場合のみ）、MPL/GPL/LGPL トリプルライセンスに再ライセンスすることができます。ただし、ソースコードに Netscape ではない他者からの著作権の許で配布されているコードが含まれる場合、私たちは、著作権保持者にそのコードのライセンスを変更しても良いかどうか、許可を得なければなりません。

そのため私たちは、MPL（または MPL/GPL）下にあるソースファイルの著作権保持者を特定し、MPL/GPL/LGPL トリプルライセンスの許でソースファイルを再ライセンスする許可を得なければなりません。この作業はある程度自動化されているとはいえ、人手が必要となり、そのため、作業が完了するには時間がかかるかもしれません。ソースファイルの著作権保持者を特定し、必要な許可を得た後は、そのファイルを前述の MPL/GPL/LGPL トリプルライセンスに変更します。

私たちはまた、現在も NPL/GPL デュアルライセンス下にある、残りのソースファイルの再ライセンスも目指しています。そのようなファイルは、NPL/GPL/LGPL トリプルライセンスに再ライセンスされることになります。

オリジナルのライセンスは、あらゆる Mozilla コードに対して効力を持ったままなのですか？

はい。GPL や LGPL と既に互換性をもつその他のライセンス（つまり、NPL、MPL でもなく、また MPL ベースのデュアルあるいはトリプルライセンスでもないライセンス）下にある、ある特定の提供されたソースファイルは、そうです。

既存の Mozilla ソースファイルの中には、（改変された）BSD License（英文）、MIT License（英文）、あるいは同様のライセンス下にあるものがあります。そのようなソースファイルの例としては、JPEG ライブラリコードがあります。私たちには、そのようなライセンス下にあるコードの再ライセンスを目指す理由がありません。なぜなら、問題のライセンス条項は、既に MPL、GPL、LGPL の全てと互換性がありますし、プロプラエタリーソフトウェアへのコードの組み込みも、（Mozilla ライセンス協定に基づいて）許可しているからです。

そのような他のライセンス下にあるコードを含む Mozilla ソースディレクトリ（mozilla トップレベルディレクトリの下）には、次のものがあります（ただし、これだけに限りません）： 

dbm
expat
gc/boehm
jpeg
modules/libimg/mng
modules/libimg/png
modules/zlib
xpcom/typelib/xpidl/

Mozilla ビルド手続きの一部として追加された makefile を除いて（これらは新しいライセンス方式を使用してライセンスされます）、これらディレクトリ内の全てのファイルは、オリジナルのライセンス下におかれたままにしておかなければなりません。

特にライセンスの注意書きがない Mozilla ソースは、どういう扱いになるのですか？

Netscape の社員によって作成されたそのようなファイルは、どれも MPL 下に置かれるものとされ、自動的に再ライセンスされます。その他のファイルに関しては、そのファイルの提供者に連絡を取り、再ライセンスの許可を要請します。

Mozilla ソースファイルの多くは、明示的にライセンス告知（または明示的な著作権告知）がされていません。そのようなファイルの再ライセンスは、次のように行われます：

問題となるファイルが Netscape の開発者によって作成されたものである場合、私たちは、適用されるライセンスは NPL（ファイルの置かれているソースディレクトリによっては、NPL/GPL デュアルライセンス）だと考え、自動的に NPL/GPL/LGPL トリプルライセンスに再ライセンスされます。

問題となるファイルが Netscape の社員によって作成されたのではない場合、適用されるライセンスは NPL（または NPL/GPL デュアルライセンス）ではないと考え、コードの提供者に連絡を取り、そのファイルの再ライセンスの許可を取ります。

再ライセンスの許可を得るには

なぜ非 Netscape 貢献者から、NPL の "特記" 事項の替わりに、もともと NLP のライセンス下にあるコードを再ライセンスする許可を得なかったのですか？

なぜなら、貢献者のコードを NLP 下でライセンスする許可を得ることで、非 Netscape 貢献者は、すでに間接的に再ライセンスする許可を与えているからです。

貢献者がもともと NPL 下でコードを提供した場合、NPL が、第三者に他のライセンス下で提供された彼らのコードを、Netscape（現在の AOL Time Warner）が再ライセンスするという特別な権利を認めていることを、貢献者は知っている（または様々な情報源から簡単に理解できる）と考えられるからです。いずれにしろ、コードを提供すると選択することで、貢献者は、彼らのコードがこのような方法で再ライセンスされることを間接的に許可しているのです。なぜなら、NPL 下でコードを提供する許可をすることで、NPL/GPL/LGPL トリプルライセンス下で再ライセンスする許可を得るための時間や労力の浪費を防ぐという観点から、既に（間接的に）任意の再ライセンスを認めているからです。

こうすることで貢献者に対して公平でいられると、私たちは信じています。なぜなら、新しいライセンス方式は、以前のライセンス下で認められてきたものと同等の権利を保証しているからです。

NLP （または MLP/GPL）下で Mozilla にコードを提供した場合、どうしてこのライセンスの変更に許可を与えなければならないのですか？

なぜなら、そのような許可を与えるということは、あなたが MPL（または MPL/GPL）下でコードをライセンスする許可を与えることとは矛盾しないからです。またそうすることで、Mozilla コードをより多くの人々に利用してもらうことができ、それが他の人の手によって Mozilla をより使いやすくすることにもつながるからです。

Mozilla コードに使用されているライセンス方式は、常に二つの特徴からなっています：ソースコードを Mozilla コードへの変更をしやすくするよう求めていること、そして Mozilla コードを他のプロプラエタリーコードと組み合わせて、プロプラエタリー製品を作成し、頒布することを認めていることです。"標準" のライセンス（NPL や MPL を何らかのかたちで使用しているライセンス）下で Mozilla プロジェクトにコードを提供している人は誰でも、結果的に、こうした一般的な条項の許に彼らのコードを頒布することを認めているのです。

新しい Mozilla のライセンス方式（MPL/GPL/LGPL トリプルライセンス）は、それまでの体系との互換性を保っているといえます：Mozilla ライセンス方式も、Mozilla コードへの変更をできるよう、ソースコードを利用可能にするよう求めていますし（これは、MPL、GPL、または LGPL 条項下のコードでライセンスを使用するように求められています）、Mozilla コードをプロプラエタリープログラムに使用することも認めています（これは MPL によって認められており、ケースによっては LGPL によっても認められています）。そのため、こうした観点から、新しいライセンス方式は、あなたが提供したコードを使用する許可を既に与えているため、先のライセンス方式と互換性があるのです。また、あなたのコードに MPL/GPL/LGPL トリプルライセンスを使用する許可を与えることで、先のあなたの許可との互換性を保てるのです。

あなたのコードを再ライセンスする許可をすることで、あなたは、Mozilla コードをより幅広いユーザーや開発者のコミュニティに利用可能にする手助けをすることにもなるのです。その結果、Mozilla に改良が施されるかもしれませんし、より多くの Mozilla を基礎にしたアプリケーションが作成され、改良され、あなたにとっても有益な新しいアプリケーションが作成されるかもしれないのです。

私が提供した Mozilla コードを MPL/GPL/LGPL のトリプルライセンスで許可を付与したい場合、どうすればいいのですか？

relicensing@mozilla.org にメールを送って下さい。

私たちは、NPL または MPL/GPL デュアルライセンス下で Mozilla ファイルにコードを提供した（非 Netscape）貢献者のリストを作成しており、彼らのコードを MPL/GPL/LGPL トリプルライセンス下に再ライセンスする許可を要請しています。もしあなたがそのようなコードを提供したことがあり、このような方法で、あなたの全てのコードを再ライセンスしても良いという全面的な許可を与えて下さる場合には、どうぞ relicensing@mozilla.org までメールを送って、あなたが提供したコードを全て MPL/GPL/LGPL トリプルライセンス下で再ライセンスするという許可を宣言して下さい。

コード提供者が自分の提供したコードに、再ライセンスする許可を与えない場合はどうなるのですか？

私たちは、他の貢献者が代わりのコードを MPL/GPL/LGPL トリプルライセンスで提供してくれるまで、そして提供してくれない限り、そのようなコードも Mozilla ソースコードの中に含めます。

前で論じたように、Mozilla ソースファイルが現時点で MPL（または MPL に基づいたデュアルライセンス）で配布されている場合で、もしそのようなファイルに Netscape によって著作権が保護されていない提供コードが含まれていたとしたら、そのコードの著作権保持者に、ライセンスを変更するための許可を求めなければなりません。その間、そうしたファイルは、そのコードの既存のライセンス下で使用することができます。

私たちの最終的な目標は、mozilla.org を通じて配布される全ての Mozilla ソースファイルを、MPL/GPL/LGPL または NPL/GPL/LGPL トリプルライセンス下（または、前述のような、その他の適切なライセンス）に再ライセンスすることです。Mozilla ソースコードに、コードの著作権保持者が MPL/GPL/LGPL トリプルライセンス下に再ライセンスすることを拒否するようなコードが含まれることがあるかもしれません。そのような場合には、私たちは、誰かが新しいコードを書いて、再ライセンスできないコードと置き換えることができるかを、調査します。（この作業によって、全く新しいソースファイルを書かなければならなくなるかもしれませんし、既存のファイルの一部を変更するだけで済むかもしれません。）

そのようなファイルが現在のライセンス下に残っている場合は、二つの結果が考えられます：問題となるファイルが、現時点で MPL（のみ）下でライセンスされている場合、そうしたファイルの中のコードは、開発者によって、 MPL またはプロプラエタリーライセンス下で Mozilla ベースの製品で作成され、配布されることで使用される可能性があります。ただし、開発者が LGPL または GPL 下でソフトウェアを作成し、配布することで使用されることはありません。もし問題のファイルが、現時点で MPL/GPL 下にライセンスされている場合、そのファイルの中のコードは、MPL、GPL、またはプロプラエタリーライセンスの下で、開発者によって Mozilla ベースの製品が作成され配布される可能性があります。ただし、開発者によって LGPL 下でソフトウェアを作成し配布するのには使用されることはありません。

再ライセンスに対する私たちの姿勢

どうして NPL を完全に廃止してしまわないのですか？

なぜなら私たちが、AOL Time Warner から NPL を廃止するためのに必要な全ての許可を得ていないからです。

より長期的には、私たちは Mozilla コードベースでの NPL を段階的に廃止していきたいと考えています。しかし、現時点では、私たちはまだ、廃止するために必要な全ての許可を得ていません。そのような許可を得た場合、私たちは、NPL/GPL/LGPL トリプルライセンス下にある Mozilla ファイルを、MPL/GPL/LGPL トリプルライセンスに再ライセンスするはずです。（例えば、NPL や NPL/GPL デュアルライセンスといったその他の NPL ベースのライセンスでリリースされたファイルは、再ライセンスの最初の段階として NPL/GPL/LGPL トリプルライセンスへと変更される予定です。）

はじめに宣言したとおり、 MPL/GPL デュアルライセンスを使用しないのはなぜですか？

なぜなら、MPL/GPL デュアルライセンスでは、開発者が LGPL で配布された Mozilla コードの中のライブ利を私用することができないからです。

ある人たちは、GPL 下でライセンスされた他のコードがあるプログラムの中で、Mozilla コードを含めることができるようにしたいと思い、また GPL 条項や条件下で派生した製品を（GPL で定められたように）配布したいと考えるかもしれません。また別の人たちは、Mozilla コードをライブラリに組み込みたいと考えたり、別のコードが LGPL 下にある別の製品に組み込みたいと考え、LGPL 条項や条件下の派生製品を配布したいと考えるかもしれません。

そこから私たちの問題が発生しました：どちらの可能性をも可能にするためには、どのような Mozilla ライセンスがあるのだろう？各方面からの意見を参考にして、私達は、以前に提唱した MPL/GPL デュアルライセンス方式が、十分ではないという結論に達したのです。もし誰かが、そのようなデュアルライセンスされた Mozilla コードをプロプラエタリーコードと組み合わせた場合（そのコードが LGPL 下でライセンスされていた場合）で、Mozilla コードのために MPL ライセンスオプションを使用した場合、MPL 下のコードと GPL 下のコードを組み合わせたときと同様の問題が発生してしまいます。その一方、そのような Mozilla コードをプロプラエタリーコードと組み合わせて、MPL/GPL デュアルライセンス下の Mozilla コードのために提供された GPL ライセンスオプションを使用した場合、派生した製品を LGPL 条項の許ではなく、GPL 条項の許で配布しなければならなくなります。

対象となる人たちが、彼らにとって重要なある理由によって（GPL ではなく）LGPL を使用することにした場合、Mozilla ライセンス方式を変更したほうが良いだろうと考えられます。Mozilla コードを組み込むことができ、しかも派生した製品に対しては LGPL を適用することができるからです。こうしたことができる方法は他にもあるとはいえ、MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスを使用することが、最も分かりやすいアプローチだと考えたのです。なぜなら、これならば、Mozilla コードを（トリプルライセンスの一部としての LGPL の定めるところとして）LGPL 条項や条件下で配布することができますし、同時に、Mozilla コードを使用したソフトウェアを（トリプルライセンスの一部としての GPL の定めるところとして）GPL 条項や条件下でも配布することができるからです。

Free Software Foundation は LGPL の使用を避けるように勧告していますが、なぜ mozilla.org はこのライセンスの使用を奨励しているのですか？

Mozilla プロジェクトの目標は、FSF の目標とは異なります。私たちは、新しいトリプルライセンスで LGPL を使用することで、LGPL を使用に従事する（または調べる）開発者の利益になると信じています。また、一方では、ソフトウェアユーザと自分たちのソフトウェアに GPL を使用する開発者太刀の自由を保障することにもなるのです。

Free Software Foundation の目標は、フリーのソフトウェアの作成を奨励することと、（フリーではない）プロプラエタリーソフトウェアの作成を阻止することです。そのため、FSF は、（フリーソフトウェアのライブラリをプロプラエタリーソフトウェアに組み込むことを可能にしている）LGPL の使用を避けるように勧告していますし、あらゆるソフトウェアの派生製品もまた同じ GPL の条項下にライセンスされるように、全てのソフトウェアを GPL 下でライセンスするように奨励しています。しかし、Mozilla ライセンス方式の目指すところは、Mozilla コードをその他の（非 Mozilla コード）と組み合わせて、Mozilla コードそのものに適用されるライセンス条項とは異なるライセンス条項下で配布できるような、"より大きな製品" を創り出すことを可能にすることなのです。

そのため例えば、mozilla.org を通じて配布される Mozilla コードは、他のコードと組み合わせて、プロプラエタリーライセンス条項下で配布できる、より大きな製品を作製することが現在できますし、将来もできます。Mozilla ライセンス方式を変更した際のもともとの意図は、Mozilla コードをその他のコードと組み合わせて、GPL 条項下でも配布できる、より大きな製品を作成できるようにすることでした。私たちは、あらゆる新しい Mozilla ライセンス方式もまた、Mozilla コードと他のコードを組み合わせて、LGPL 条項下で配布できる、より大きな製品を作成できるようにすることにあります。私たちは、そうして出来上がったより大きな製品がプロプラエタリーライセンス、または GPL 条項下のどちらかで配布されることを正当化できませんし、一方では、そうしたより大きな製品が LGPL 条項下で配布されることを阻止することも正当化できません。

実際に MPL または NPL 下にある Mozilla コードが既に、LGPL 下のコードと組み合わされて、その結果できるより大きな製品が LGPL 条項下で配布される場合は、全く問題ありません。もしそのような事例があれば、私たちは新しい Mozilla 体系の中で LGPL のための特別条項を作る必要はありません。しかし、LGPL が示すところによると、もし MPL が（FSF が主張するように）GPL と互換性がないならば、MPL は LGPL とも互換性がないことになってしまいます。同様のことが NPL にも当てはまります。そのため私たちは、新しい Mozilla ライセンス方式が、GPL と共に LGPL に対応しなければならないという結論に達したのです。

ユーザーに対して認められている自由の観点からは、MPL/GPL/LGPL トリプルライセンス方式は、当初考慮されて後に破棄された、MPL/GPL デュアルライセンス方式と互換性があります。（これはまた、NPL/GPL デュアルライセンスと比較したときの、NPL/GPL/LGPL トリプルライセンスにも当てはまります。）もともとの MPL/GPL の案のように、提案された MPL/GPL/LGPL トリプルライセンス方式では、ライセンス取得者が望めば、ノンプロプラエタリー派生製品を、GPL のみのライセンス下で作成することができます（例えば、そのような製品をプロプラエタリー製品や、またはフリーではないソフトウェアとの組み合わせで使用することを阻止することです）。MPL/GPL/LGPL ライセンスではまた、もしライセンス取得者が望めば、ノンプロプラエタリー派生製品を、LGPL のみのライセンスで作成することができます。

ライセンス取得者が、 GPL のみ（または LGPL のみ）Mozilla コードベースのバージョンを作成する自由があるとはいえ、mozilla.org は、Mozilla コードを他の LGLP または GPL 下のコードを使用するライセンス取得者が、MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスの条文を保持し、GPL または LGPL 条項と共に、MPL（または NPL）条項下で、 ライセンス取得者が作成した Mozilla コードの使用を認めます。同様に mozilla.org は、ライセンス取得者が、Mozilla コードを MPL または NPL（または、GPL または LGPL 以外の他のライセンス）下のコードと使用する際、元の MPL/GPL/LGPL（および NPL/GPL/LGPL）トリプルライセンスの条文を保持すること、そして MPL（または NPL）下と共に GPL または LGPL 条項下でライセンス取得者が作成した Mozilla コードを使用できるようにすることを求めています。

私たちがこうしたことを求めるのは、Mozilla のコードベースが誰もが利用できる共有の資源としてあるべきだと信じるからです。そして、人々が全ての人の利益のために、この共有されたコードベースを後押しして欲しいのです。この目標を達成するために、私たちは、新しいソースファイルが MPL/GPL/LGPL トリプルライセンス条項下にない限り、そのファイルを mozilla.org がホストする Mozilla ソースコードリポジトリに受け入れることはありません。（場合によっては、mozilla.org が MPL、NPL、LGPL、および GPL の全てと互換性があるその他のライセンス下にあるコードを、受け入れる可能性もあります。mozilla.org はまた、LGPL のみの許にあり、LGPL に定められたような、他の非 LGPL コードと共に使用されたライブラリを形成するコードを受け入れる可能性もあります。しかし、mozilla.org はそのようなライセンスを承認しなければなりませんし、そのようなコードは、Mozilla コードの残りではなく、隔離された場所に保管されることが求められる可能性があります。）

特に、私たちは LGPL または GPL のみでライセンスされたコードの提供を受け入れることはないでしょうし、（再ライセンスが完了したら）MPL または NPL のみでライセンスされたコードの提供も受け入れることはないでしょう。（現時点では、Mozilla の貢献者は、NPL 下で、またはその他のライセンスやその組み合わせでライセンスして、新しいソースファイルを作成してはいないでしょう。貢献者は、そのようなコードが、まだ再ライセンスされていない既存のファイルへのパッチである時にのみ、MPL か MPL/GPL デュアルライセンスを使用するべきです。）

MPL/LGPL デュアルライセンスを適用しないのはなぜですか？LGPL は GPL 条項の許でも使用が許可されているのではありませんか？

MPL/LGPL デュアルライセンスを使用することで、事例によっては、人々に Mozilla ソースファイル上のライセンス告知の変更を要求することになり、GPL 条項の許で Mozilla コードを配布したいと考えている人々に不必要な不便をかけてしまいます。MPL/GPL/LGPL トリプルライセンスでは、こうした潜在的な不便さを減らすことができます。

ライセンス方式の一つの可能性としては、MPL または LGPL のどちら下の条項と条件のもとに Mozilla コードを認めるという、デュアルライセンスでした。言い換えると、GPL を明示的に代替ライセンスとしては含まず、LGPL によって可能となる GPL 条項の許で使用するというものです。しかし、問題を検討した結果、私たちは、このアプローチが、Mozilla コードを GPL 条項の許で配布しようとする人々に、不必要な不便を引き起こすことになるという結論に達しました。

例えば、MPL/LGPL デュアルライセンス下にある Mozilla ソースファイルを考えてみましょう。このデュアルらい戦では、MPL 条項かまたは LGPL 条項の許でコードを使用できるよう許可しています。そして、ある開発者がアプリケーションを開発したいと考えました。そのアプリケーションでは、Mozilla ソースファイル内のルーチンが、GPL 条項下にある別のソースファイル内のルーチンを呼び出します。そのようなアプリケーションは、全体として GPL 条項の許で配布されなければなりません。そのため、その開発者は、そのアプリケーション全体の一部分としての Mozilla コードを、GPL 条項の許で配布する許可を得なければなりません。

MPL/LGPL デュアルライセンスの許であれば、その開発者は、そのコードを GPL 条項の許で配布するための明示的な許可を必要とはしません。明示的な許可が必要な場合は、LGPL 条項の許でコードを配布する際のみのこととなります。そのため、その開発者は、デュアルライセンスの LGPL 部分を利用しようと考えるでしょうし、GPL 条項の許での（再）配布を許可している LGPL 内の条項を有効利用することができます。

LGPL が GPL 条項の許で配布を認めるのは、二つの場合があります。一つは、LGPL の第 6 節を利用するものです。この節は、LGPL 条項下で配布された "ライブラリ" を、非 LGPL 条項の許で、ライセンス条項が製品全体への変更を認め、必要な条件を身対している場合には、"ライブラリ" を "ライブラリを使用する製品" の全ての部分または一部分として（再）配布することを認めています。GPL が問題の条件を満たしていることから、この条項によって、仮定的な MPL/LGPL デュアルライセンスの許、Mozilla コードで構成されるライブラリを、GPL 条項下のコードからなるアプリケーションと共に配布することができることになります。そのデュアルライセンスによって、Mozilla コードが LGPL 条項の許で配布できるようになり、LGPL の第 6 節は、GPL 条項の許で製品全体を配布することを可能にしているのです。

しかしこれは、私たちの例には当てはまりません。なぜなら、Mozilla コードは、LGPL が定義するような "ライブラリ" として存在するのではないからです。そうではなく、Mozilla ソースファイルの中のルーチンは、GPL 下でライセンスされた、（それらルーチンによって呼び出されたものとは反対に）ソースファイルの中のコーリングルーチンなのです。そのため、私たちの例の中の開発者は、LGPL の第 6 節を利用できないため、GPL 下で配布を認めた LGPL の他の条項を使用しなければなりません。それは、第 3 節です。LGPL の第 3 節は、明示的に LGPL 条項下で配布されたコードを、替わりに GPL の条項下で（再）配布することを認めています。しかし、これを行うためには、開発者は、Mozilla ソースファイル内のライセンス告知を実際に変更して、GPL ライセンス告知に置き換えなければなりません。

仮定的な MPL/LGPL デュアルライセンスの条項は、この切り替えを認めていますが、私たちの例の中の開発者が切り替えを行うのは不便です。特に問題の Mozilla コードが、単一のソースファイルから成っておらず、何十も、何百も、時には何千ものソースファイルから成り立っている場合には。そしてこの不便さは、不必要なものでもあります。つまり、Mozilla ソースファイルが、GPL 条項下のコードの使用を明示的に認めている場合、ライセンス告知を変える必要は全くなくなるのです。

そのため私たちは、MPL/GPL/LGPL トリプルライセンスを、（NPL/GPL/LGPL トリプルライセンスを NPL 下の既存のコードのためのライセンスと組み合わせるかたちで）利用するというアプローチを取ることにしたのです。このライセンス方式は、（MPL および LGPL 条項と共に）GPL 条項下で利用するために、明示的に提供されたもので、前述のようにライセンス告知を変更する必要はありません。

トリプルライセンス方式を適用するのではなく、 MPL 自体を改定して GPL および LGPL との非互換性を明確にしないのはなぜですか？

MPL そのものを変更するというのは、時間と労力を必要とします。それに Mozilla プロジェクトは（他のプロジェクトと同様）、複合ライセンス方式を、深刻な問題もなく既に使用しています。

トリプルライセンスを使用するのではなく、別のアプローチがあるとすれば、明示的に GPL（そして LGPL）との互換性問題を取り扱う条文を含む、MPL（および NPL）の新しい版を作成することでしょう。（FSF は以前、そのようなアプローチを、NPL と GPL との非互換性問題に言及する一つの方法として提案していました。）しかし、この問題を検討した結果、私たちは、MPL や NPL を変更することが、時間の浪費と潜在的に問題を引き起こしやすくなるだろう、という結論に達しました。

MPL を変更することで、Mozilla プロジェクトとは独立した立場から、自分たちのコードを使用するために MPL を採用している開発者に影響を与える可能性もあります。そのような開発者たちが、新しい MPL の変更を認めず、明示的に MPL の古いバージョンを使用せざるを得なくなるかもしれませんし、彼ら独自の MPL ベースのライセンスを作成することになるかもしれません。（または、こうならないように、私たちは、開発者が条文を取り入れるかどうかを指定できるように、MPL 内の条文をオプショナルにしなければならないかもしれません。）最後に、MPL を変更することは、Open Source Definition（英文、日本語版）との関連から、証明のために Open Source Initiative（英文）に変更したライセンスを提出しなければなりません。これはまた時間の浪費につながります。

そのため私たちは、より良い方法とは、単純にトリプルライセンスを使用することだ、という結論に達しました。このアプローチは、以前 Mozilla コードに使用していたデュアルライセンスアプローチの、自然な形での拡張で、特に深刻な問題もなく他のプロジェクトでも使用できます。既にオープンソースおよびフリーソフトウェア開発者にとって馴染み深いライセンス（MPL version 1.1、NPL version 1.1、GPL version 2、および LGPL version 2.1）を使用することにつながり、他の新しく異なるライセンスの導入が必要なくなります。

他のあらゆるライセンスと互換性のある非 copyleft ライセンス （例えば MIT や BSD ライセンス）の許で Mozilla コードを再ライセンスしないのはなぜですか？

歴史的に見て、Mozilla コードは常に何らかのかたちの copyleft ライセンスの許でリリースされてきました。そして私たちは、Mozilla コードへの変更を共有するように働きかける copyleft 条項を使用し続けたいと考えています。

NPL や MPL を作成した背景にある動機の一つは、Mozilla コードを使用する全ての当事者間で、変更の共有を奨励することでした。BSD license（英文） や MIT license（英文） 又は同様の非 copyleft ライセンスを使用することを検討したこともありましたが、最終的に NPL/MPL copyleft 機構を採用することにしたのです。オリジナルの Netscape Public License FAQ から引用すると：

Netscape の見解によると、BSD ライセンスでは、開発者が Communicator ソースコードへの彼らの変更を、コミュニティに還元することを、確実なことにはできません。我が社は、ソース開発努力の長期的な有効性を確実にすることが重要だと考えているのです。

私たちは、この目標は推進する価値があると信じます。そのため、私たちは、copyleft 条項の使用をそのまま保持することにしたのです。つまり、新しい MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスの許で Mozilla コードｗのために三つのライセンスオプションが利用可能となり、全てのライセンスオプションには copyleft の特徴が、（異なる段階で）備わっており、Mozilla コードへの変更は、そのコードを使用した製品を配布する際に、配布することが求められています。

開発者への影響

この新しい Mozilla ライセンス方式は、 Mozilla コードを利用してプロプラエタリーアプリケーションを作製し、配布したいと考えている開発者にどのような影響を与えるのでしょう？

全く影響はありません。Mozilla コードを取り入れてプロプラエタリーソフトウェアを作成し配布している開発者は、MPL または NPL 条項や条件の許で、そのまま活動ができます。彼らが今までずっと行ってきた通りのままです。

MPL（NPL も同様に）はもともと、MPL 条項下で配布されているソースファイルをその他のライセンス下のソースファイルと組み合わせて配布できるようにし、その結果できる製品を非 NPL（または非 NPL）ライセンス下で配布できるように、デザインされています。（この権利は、MPL または NPL が満たされるという要求を満たすことが条件となっています。例えば、作成された製品をユーザー向けに配布する場合、配布者はまた、MPL または NPL 下で配布されたソースコードから作成されたコード部分をも、ユーザに対して利用可能にしなければならないのです。）

MPL や NPL のこの特徴で、開発者は、Mozilla コードを組み込んだプロプラエタリー製品を配布できるのです。（Netscape 6 はそのような製品の一例です。）MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンス方式の許では、開発者は、MPL（または NPL）条項下の Mozilla コードを使用し、Mozilla コードを基礎にしたプロプラエタリー製品のそうした部分に対しての MPL（または NPL）の様々な要求を満たしながら、Mozilla ベースのプロプラエタリー製品を作成し、配布し続けることができるのです。

（理論的には、開発者はまた、もしその Mozilla コードがライブラリの形であり、その開発者が LGPL の関係する要求に準じていれば、Mozilla ベースのプロプラエタリー製品を LGPL 条項の許で配布することができます。）

この新しい Mozilla ライセンス方式は、 Mozilla コードを利用してソフトウェアを作製し LGPL の許で配布したいと考えている開発者にはどのような影響を与えるのでしょう？

新しいライセンス方式が完全に実施された場合、LGPL 下で配布されたライブラリや関連ソフトウェアを作成している開発者は、LGPL 条項及び条件で、そのソフトウェアの中であらゆる Mozilla コードを使用することができるようになるでしょう。

全ての Mozilla ソースファイルが MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンス下で再ライセンスされた場合、開発者は、Mozilla ソースファイルを LGPL 条項下で使用するオプションを利用することで、また Mozilla ソースファイルからのコードを基礎にしたソフトウェアのそうした部分に対しての LGPL の様々な要求に準じることで、Mozilla ベースのライブラリーや関連するソフトウェアを LGPL 下で配布することが可能となります。

MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスの許では、開発者はまた、既存のソースファイルにある LGPL ライセンス告知を変更することなく、Mozilla コードの一部分を取り出して、既に LGPL にある既存のソースファイルの中で使用することもできます。（これが可能なのは、トリプルライセンスの鋳型条文が、告知の MPL または NPL 部分を削除し、LGPL 告知の部分のみを使用することを認めているからです。）

全ての Mozilla ソースファイルが MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンス下に再ライセンスされなくても、LGPL の許でライブラリを配布している開発者は、MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスの許で既に再ライセンスされている、あらゆる Mozilla ソースファイルを使用することができることを覚えておいて下さい。

この新しい Mozilla ライセンス方式は、 Mozilla コードを利用してソフトウェアを作製し GPL の許で配布したいと考えている開発者にはどのような影響を与えるのでしょう？

新しいライセンス体系が完全に実施された場合、GPL の許で配布されるソフトウェアを作成している開発者は、そのソフトウェアの中で GPL 条項および条件下で、あらゆる Mozilla コードを使用することができます。

全ての Mozilla ソースファイルが MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンス（またはその他の互換性のあるライセンス）の許で再ライセンスされた場合、開発者は、GPL 条項の許で Mozilla ソースファイルを使用するオプションを利用することで、GPL 下で Mozilla ベースのソフトウェアを作成し、配布することができます。これにより、開発者は、製品全体を GPL 条項の許で配布することができるようになり、同時に、Mozilla ソースファイルからのコードに基づいたソフトウェアのそうした部分のための GPL が定める要求に応えることができるのです。

MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンスの許では、開発者はまた、ソースファイル内の既存の GPL ライセンス告知を変更することなく、Mozilla コードの部分を抜き出し、それらを既存の GPL 下のソースファイルに使用することができます。（これは、トリプルライセンスの鋳型条文が、告知の MPL または NPL 部分を削除し、GPL に関連する告知で代用することを認めているために、可能となることです。）

あなたが Mozilla コードの一部を GPL 条項下で配布したい場合、全体の再ライセンスプロジェクトが完了していなくても、配布できることを覚えておいて下さい。再ライセンスは段階的に行われるため、あなたが使用したいと思う特定のファイルを確認することを忘れないで下さい。なぜなら、そうしたファイルは、既に MPL/GPL/LGPL（または NPL/GPL/LGPL）トリプルライセンス下にあるかもしれませんし、過去に MPL/GPL または NPL/GPL デュアルライセンスの許に置かれていたかもしれないからです。
