Goal

このドキュメントの目的は、どのようにして Mozilla ブラウザクライアントを 局地化すれば良いのかの短いイントロダクションを与えるものである。 そのため、読み手は 彼らの好きな言語での Mozilla の局地化についてだけの 長いドキュメントを調べる必要がない。
 

ローカライズには何が必要か

Mozilla では、全ての UI リソースはインストールディレクトリの "chrome/" ディレクトリに置かれている。 ローカライズ可能なファイルは "packageType/locale/locale_name" の 配下にある。 "packageType" は "addressbook", "bookmarks", "navigator", etc...  のような UI コンポーネントの一つである。 そして、"locale_name" は Mozilla がローカライズされる言語である。

Mozilla にはローカライズされるために必要な３つのタイプのファイルがある。:

  1. DTD (*.dtd) ファイル。 この ドキュメント は、私たちの XUL ファイルのローカライズ戦略を説明する。 一般的な DTD ファイルはローカライズに必要なエンティティのリストを含む。 エンティティ宣言はこのように見える。:

    2.  
      <!ENTITY browserCmd.label "New Browser Window">


    The entity with name/key, "browserCmd.label", has the entity value, "New Browser Window". エンティティ値はローカライズされるために必要なものである。 エンティティ名の記述 "browserCmd.label" は自身を説明している 。: それは "browserCmd" メニューアイテムのラベル文字列である。 XUL の記述者またはローカライズエンジニアもまた、 多くのコメントとして DTD ファイルの中に含めようとするだろう。 それらのコメントはエンティティ宣言の右上に配置され、
     

      <!-- LOCALIZATION NOTE  entity_name: comments -->
    のようにマークされる。


    例えば,
     

        8<!-- LOCALIZATION NOTE  throbber.url: Do NOT localize all the URL -->
        9 <!ENTITY throbber.url "http://www.mozilla.org">


    Mozilla では、XUL/RDF/XML ファイルの UTF-8 が標準のドキュメントのエンコード方法 であることを書いておく。 テキストの宣言セクションの中で別に示されない限り 翻訳された文字列は UTF-8 エンコーディングになるだろう。 EUC, SJIS やその他のネイティブなエンコードは、ユーティリティプログラムによって UTF-8 にコンバートされることができる。 nsconv(.exe) はapprunner(.exe)と同じディレクトリの中に提供されている。 The usage:
     

      nsconv -f source_encoding -t target_necoding source_file target_file
  2. プロパティファイル(*.properties)。 プロパティファイルは 文字列リソース バンドル機構のデータストレージとして使用される。 プロパティファイルの中の文字列リソースは、

    3.  
      name=value

      フォームの中に現れる。


    "value" 部分は、ローカライズに必要なものである。 Localization notes もまた、必要な場所に追加される。例えば、
     

       25 #-----------------------------------------------------------
       26 #   LOCALIZATION NOTE  InstallFile:...
       27 #-----------------------------------------------------------
       28 InstallFile=Installing: %s
       29 ReplaceFile=Replacing: %s


    "escape-unicode" だけが、プロパティファイルで受け入れられる ドキュメントエンコードだということを覚えておくように。 あなたは、翻訳の間、あなたの好きなエンコードを使い、 そして、多くの Unix システムで動作するユーティリティである "native2ascii" を通して escape-unicode へ変換することができる。 Java SDK もまた、"native2ascii" を含んでいる。 加えて、nsconv(.exe) は ネイティブ・エンコードを escape unicode に より早く変換するために使うことができる。 escaped-unicode のために、 "x-u-escaped" の名前を使いなさい。
     

  3. chrome レジストリ は "chrome/registry.rdf" 。 ローカライズされたパッケージタイプのファイル群は、 "locale_name" に関連づけられたサブディレクトリに置かれる。 Mozilla クライアントは、それに従ってローカライズされたファイルを 拾い上げ、chrome レジストリの中の名前エントリ(in-between <name>...</name>) は、"locale_name" で置き換えられなければならない。(10/29/99) レジストリファイルと コンフィグレーション可能な Chrome は 再構築中であることを 覚えておくように。 chrome レジストリのコンフィグレーションを行う新しい方法は それが決定した後にポストされるだろう。 今は、あなたは "en-US" ディレクトリの中のファイルをローカライズし、 後で、それらを再配置することができる。

Notes

  1. 言語パックは、プラットフォームをまたがって適用される。
  2. mozilla-win32.zip  は、実行可能な Mozilla、英語ファイル、ボスニア語ファイルを含んでいる。 デフォルトで、UI 言語はボスニア語である。
  3.   bosnian.zip はボスニア語のために必要なファイルのみ含んでいる。 あなたはそれらを chrome/ ディレクトリに unzip し、 chrome/registry.rdf の中のこれらの UI 文字列をピックアップして、 全ての "en-US" 名前エントリを "da-DK" に変更しなければならない。
  4.   M8.5 の時点では、いくつかのウィンドウが 特別な UNICODE キャラクタが使用されて いた時にクラッシュするため、 bosnian-BA DTD ファイルの中では、ASCII キャラクタだけが使用されている。 私は Mozilla が可能な限り上手く動作することを望み、 それで、私は全ての ASCII one の中に我々の特別な文字に変換した。 この方法はここに割り当てられて使われている。 我々のキャラクタを表示させるために、人々は Windows 9x or NT の PANEURO バージョンを得る必要がある。 我々のソフトウェアは、大抵、この特別なキャラクタ無しである。 次のリリースのために、この変更を計画する。
  5.   この zip ファイルは、デンマーク語のために必要な言語ファイルのみ含んでいる。 あなたはそれらを chrome/ ディレクトリに unzip し、 chrome/registry.rdf の中のこれらの UI 文字列をピックアップして、 全ての "en-US" 名前エントリを "da-DK" に変更しなければならない。

ツリーへのアクセス権限の取得

局地化の貢献者と翻訳者は、Mozilla への局地化の作業の中で、 チェックインするために、 CVS アクセス権限を必要とするだろう。 これはツリールールと、どのようにして CVS アカウントを得るかを 説明したページへのリンクである。
 http://www.mozilla.org/hacking/
局地化に関連したファイル便宜を計るが、 他のモジュールのビルドプロセスの破壊を避けるため、 それらは、Mozilla ツリー mozilla/" の下に作られたディレクトリ "l10n/" に納められるだろう。 詳細は次のセクションを見なさい。
 

局地化関連作業のためのディレクトリ構造

以下のツリー構造は、 Mozilla ツリーの中での全ての局地化関連作業の便宜を計っている。
  1. ディレクトリ構造 Mozilla ツリーを伴うが、 独立したディレクトリの下の局地化関連のファイルは置かれていない CVS リポジトリを共有しなさい。 独立したディレクトリの下の局地化関連のファイルを置いてではない。 Tague Griffith は過去に、オープンソース L10n ツールの便宜を計るために 新しいディレクトリ "l10n/" を作った。 ツールはサブディレクトリ l10n/tools/" の配下にチェックインされるだろう。 我々は、ツールと同列なサブディレクトリ l10n/lang/" に置かれた 言語を定義するファイルを持つことができる。 そして、そこは別のサブディレクトリ "l10n/builds" で、それは 様々なビルド関係ファイルに便宜を計る。  ディレクトリ構造はこのように見える:

    2.  

     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     

    mozilla/

      [other modules]
      [...]
      l10n/
        tools/
          mozexptool/
          mozillatranslator/
          etc...
        lang/
          addressbook/locale/
                bosnian-BA/
                da-DK/
                da-DK-vendor1
                ja-JP-vendor1
                ja-JP-vendor2
                etc...
          bookmarks/locale/
                bosnian-BA/
                da-DK/
                da-DK-vendor1/
                ja-JP-vendor1
                ja-JP-vendor2
                etc...
          [other package types, etc...]
        builds/
          [release team's stuff]
  2. Pulling the tree. 現在の Mozilla ビルドプロセスは "l10n/" を取り寄せていない。 その結果、他のモジュールデベロッパは、彼らが明確に "l10n/" を そのようにしない限り、取り寄せないだろう。 これは、ツリーの取り寄せ時間と無駄なディスク領域の問題についての関係に ついて述べている。しかしながら、開発サイクルの間、我々はまだ、 デフォルトで l10n/ を取り寄せることを選択している。

    3.  
  3. The build script. 局地化されたクライアントの build script は、 デイリー・ビルドの一部としてセットアップされたり、 スタンドアロンスクリプトとして動かされることができる。
このアプローチの優位性を、この後に述べる。
  • l10n が 毎日の Mozilla ビルド対象の一部ではなかった時から、 ローカリゼーション・チェックイン("mozilla/l10n" の下に)は Tinderbox ルールには制限されておらず、その結果フックされていなかった。

    •  
  • ネット開発者は、アクセス権限を認められ、 直接、オープンソースツリー上で作業することができる。
"mozilla/l10n" ディレクトリを乱用することを 人々に推奨しているわけではないことを、覚えておいてもらいたい。 全ての CVS チェックイン・ルールはまだ有効である。 同じ寄贈者への礼儀や、モジュール・オーナーシップへの尊敬が、 オープンソース・コミュニティの中で必要とされている。

  • Netscape ブランドのバージョンはどの言語に対してローカライズされていくのか？
      •  
    Netscape ブランドのバージョンは少なくともドイツ語、フランス語、日本語に ローカライズされるだろう。他の言語のための将来の計画は Netscape の判断によりアナウンスされるだろう。
       
  • Netscape は Netscape ブランド バージョンの中で、現地語化の作業を Mozilla オープンソース・ツリーに対してリリースしようとしているのか？ OEM とは何だろうか？
      •  
    Netscape は、ローカライズされたファイルを、 我々がそれらを利用可能なものとして現地語化する言語の中で、 mozilla の中にポストするだろう。
       
  • 我々は、寄贈された翻訳を Netscape ブランド・バージョンの中に組み入れるだろうか？
      •  
    現在、Netscape とそのパートナーは、言語バージョンを生産しており、 我々はそのビジネス戦略をサポートする必要がある。 もし、それらの戦略が変更されたら、Netscape は mozilla.org での作業を 終了したように見えることを選択するだろう。
     
  • 我々は、どのように、同じ言語の Netscape バージョンと Mozilla バージョンの衝突を論じれば良いのだろうか？
      •  
    Mozilla と Netscape ブランドの製品は、２つの違うプロダクトになるだろう。; オープンソースデベロッパは、Netscape ブランドの製品と衝突しない、 彼らの mozilla ツリーにチェックされたローカライズされたファイルを持っている。
