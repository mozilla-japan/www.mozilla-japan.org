M13 リリースノート

インストールする前に読んでください

AOL 4.0 かそれ以前のバージョン と Mozilla のビルドを同時に走らせることは 問題の原因になるかも知れない。AOL 5.0 と Mozilla を走らす事による問題は報告されていない。

もし起動時にクラッシュしたなら あなたのMozilla レジストリを削除しなさい。 Mozillaのレジストリーは全てのプラットフォームでM9より変わっています。

  • Mac OS: システムフォルダー内のPreferencesフォルダーを開いて、 "Mozilla Registry" ファイルを消去してください。
  • Windows:Windowsディレクトリーを開いて、"mozregistry.dat"を消去してください。
  • Linux:~/.mozillaディレクトリー内の"registry"ファイルを消去してください。 file.

さらに詳しく知りたい方は、 news://news.mozilla.org/37E8A320.D3420962%40netscape.comをご覧ください。 Win32にインストールしている場合には、インストール 手順の 起動時のクラッシュに関する追加情報を読んでください。

パスワードを記憶する機能には不具合があります。 (Mac OS のみ)。最初にこの機能をonにするように求められた場合には、 キャンセルボタンを押すことをお勧めします。また、Edit:Preferences:Advanced で、 ユーザー名とパスワードの自動補完機能のチェックボックスをオフにすることでも、 この機能を使えなく出来ます。(バグ # 21596をご覧ください。)

空のディレクトリーにインストールするか、インストール前に、 前のビルドを消去してください。以前のマイルストーンやプロダクトに上書きで インストールすると、問題が生じるかもしれません。ソフトウェア開発中には、 マイルストーン間での不整合が生じる可能性があります。

（RedHat5.2のように）glibc 2.0を使うLinuxでは 、 dl関数 (dlopen(), dlsym())がthread safeでは無いという問題があります。 この問題はM13の起動時にランダムにクラッシュを引き起こす可能性があります。もしこの問題が 起こったら、何度かアプリケーションを立ち上げ直してみてください。ftp.mozilla.orgにある ソフトはglibc2.1を使う RedHat 6.0上で構築され、このような問題は生じていません。 詳しくは、バグ 8849をご覧ください。

Java Plugin 1.3は使用しないでください。これは最近の ほとんどのMozillaのビルドで動作せず、クラッシュします。 *npjava*.* というファイルが、bin/pluginsディレクトリーにも、 NS4.x のプラグインディレクトリーにも無いことを確認してください。 この問題は現在処理中です。このバグに関して詳しくは、 バグ 5369をご覧ください。

Proxiesは、このリリースで動作します。 設定の詳細に関しては、バグ8559 をご覧ください。


M13 リリース・ビルドのインストールする

インストール手順 -- Unix

  1. "mozilla"というディレクトリーを作り、そのディレクトリーにtar.gzファイルを 移してください。

    2. mkdir mozilla ;
    mv mozilla*.tar.gz mozilla

  2. mozillaディレクトリーに移り、アーカイブファイルをuntarしてください。これにより "package"というディレクトリーが作成されます。

    3. cd mozilla ;
    gzip -dc mozilla-i686-pc-linux-gnu-M13.tar.gz | tar -xvf -

  3. "package"ディレクトリーに移ってください。

    4. mv mozilla*.tar.gz ../ ;
    cd package

  4. mozillaを実行スクリプトから起動してください:

    5. ./mozilla

重要: もしシーモンキー(Seamonkey)をSolaris上で実行するには、次の Mozilla on Solaris FAQ, も読んでください。ここにはビルドと実行に必要なSolaris 2.6パッチに関する記述があります。

Unixでのメールの使用手順

すでに使用している4.xのプロファイルデータから移行することもできますし、あたらしいプロファイルを作り、 アカウントの設定（Account Setup）を使ってサーバー情報を設定することも出来ます。

インストール手順 -- Win32

WinZipが必要になるかもしれない。これはシェアウェアで、 http://www.winzip.com (あるいは http://www.download.com) から入手することが出来ます。

重要: Seamonkey インストーラー (mozilla-win32-installer-M13.exe; これは で 入手できる) をインストールする際には、必ず1度は実行しなければならない。このインストーラーは、 ２つのマイクロソフト社製のCランタイムライブラリーを更新します。これはWinZipでは更新することができません。 このライブラリーはMSVCRT.DLLとMSVCIRT.DLLで、両方ともversion 6.0.8168.0かそれより新しいヴァージョンでなければなりません。 これらの更新されたDLLファイルは、シーモンキー(Seamonkey)の起動時のクラッシュを防ぐために必要です。注意：このインストーラーは 更新した際にリブートすることを要求してきます。

  1. mozilla-win32.zipへのリンクを一回クリックしてください。ファイルは デフォルトのフォルダーにダウンロードされます。

  2. mozilla-win32.zipファイルをクリックするとWinZipが起動します。

  3. WinZipのシェアウェアの条件に"I Agree （了解）"をクリックした後、 WinZipのツールバーにあるExtract（解凍）ボタンを押してください。 x86フォルダーにアプリケーションと関連ファイルが解凍されます。

  4. プログラムを起動するためにx86フォルダーよりMozillaを選択してください。

    • ブラウザーとメールを使用するには"mozilla.exe" を起動してください。

    • geckoレイアウト・エンジンのみを使用するには、"viewer.exe"を起動してください。

注意: もし移行されていない4.xと同様に5.0の既存のプロファイルがあり、そして mozillaコマンドを実行してもシーモンキー(seamonkey)が起動しない場合には、mozilla -installerコマンドを 実行してプロファイル・マネージャーを立ち上げ、プロファイルを選択します。

インストール手順 -- Mac OS

もしStuffIt ExpanderTM がインストールされており、お使いのブラウザーが それを使うように設定されている場合:

  1. mozilla-mac.sea.binへのリンクを一回クリックしてください。ファイルが デフォルトのフォルダー（通常はデスクトップ）にダウンロードされ、StuffItTMが起動されて mozilla-mac.seaに変換されます。それからStuffItはmozilla-mac.seaを開けて、 mozilla-macフォルダーに変換してくれます。

  2. mozilla-macフォルダーを開き、"mozilla."ファイルを2回クリックしてください。 (注意: この段階で.binと.seaファイルは不要になります。）

注意: もしステップ１の際にブラウザーが何をするかを質問してきた場合には、 StuffItTMの設定がされていないのが原因です。 このような場合には、保存(Save)をクリックして下記の#3に進んでください。

もしStuffIt ExpanderTMがインストールされていないか ブラウザーが適切に設定されていない場合:

  1. StuffItTMがハードディスクのどこかにあることを確認してください。

  2. mozilla-mac.sea.binへのリンクをクリックしてください。

  3. "Save this link as..."（リンクを名前をつけて保存）を選択しなさい。

    4. ファイルダイアログが現われ、どこにファイルを保存するか を選択することができます(例えばデスクトップ)

    注意: Macのインストーラーのアクセス・パスのどこにもスラッシュを(/)使わないでください。 スペースは使用してかまいませんがその他の特別な文字は避けてください

  4. Save（保存）ボタンをクリックしてください。

  5. 新しく作られた"mozilla-mac.sea.bin"アイコンを2度クリックしてください。

  6. StuffItTMが起動し、同時に"mozilla-mac.sea"ファイルと "mozilla-mac."という名前のフォルダが作成されます。

  7. mozilla-macフォルダーを開き、"mozilla."を2回クリックしてください。(注意: この段階で.binと.seaファイルは不要になります。)

いくつかのキャッシュのバグが報告されています:

  • 時々、throbber(読み込み中のアニメーション)が止まらない. (bug 21162)
  • キャッシュを消去(clear)したり、サイズを変更したり、どのくらいの頻度でサイトのデータとキャッシュが検証されるかを 変更することが出来ない。 (bug 20893)
  • 再読み込み、およびShift-Reloadがキャッシュの動作を変更できない。 (bug 21137)

Linux での小さいフォント

  • もしフォントがあなたの環境で小さすぎるのなら、それは軽減できるかも知れない。 それはMozillaの設定ファイルにセットする方法である。preferances ファイル (~/.mozilla/Personal/prefs.js)に、次のように付け加えなさい:

    • user_pref("browser.screen_resolution", 120);

    これは dpi を 120 にセットする。もし諸君がこれを(0)に設定すると、 Mozilla は X server の dpi値を使う。もし設定が行われなかった場合には、デフォルトの値 96が使われる。

    X server のDPIの値の設置にはいくつか情報がここにある http://www.mozilla.org/unix/dpi.html.

    もし設定を正しく設定した後もフォントがまだ小さい(あるいは大きすぎる)のであれば、 次のようにしてフォントサイズを設定することが出来る:

    setenv GECKO_FONT_SIZE_FACTOR 1.5

    文字がフォントだけに影響するわけではないことに注意しなさい。スクリーンの解像度の設定が 絶対サイズの指定のすべてに影響を与える(例えば CSS の "pt" points)。

M13におけるフォント設定に関する新たな注目点を挙げておきます。 フォントに関する作業はまだ終了していませんし、ここで挙げた仕様については今後 変更される可能性があります.:
    • フォントは5つの一般的なフォントファミリー: serif, sans-serif, monospace, cursive and fantasyに設定されています
      • モジラでは "Times", "Courier",and "Arial" のようなあまり一般的でないフォントファミリー名を 対応するCJK フォントにマッピングするようなことはしません. ウェブページデザイナは、自分の思い通りの表示を求めるならば一般的なフォントを使うべきです.
    • フォント設定はCommunicator 4.xで行われていた文字コードセットによるものよりも、言語/文字 グループ (ユニコードは除きます)によるものの方が原則的です.
    • フォントサイズは: variable あるいは fixedに設定可能です.
      • variable 設定はデフォルトのHTML表示 に対してのみ有効であり、以下のように指定します:
        • user_pref("font.default", "sans-serif"); -- デフォルトはserif体になっていますからHTMLのデフォルト表示を sans-serifにしたいのでない限りこの設定をしない方がいいです. このHTML表示のデフォルトにはserifもしくはsans-serifが 利用されているほとんどでしょう.
      • fixed 設定は monospace フォント設定にしか影響しません. e.g. メールのメッセージ表示.
      • フォントのサイズはポイントではなくピクセル で表されます.
    • ここで挙げている設定については、アクセスしたページで利用されている特定のフォントがない場合にフォントを選択する際の参考になるでしょう.
    • 非アスキーのフォント名はサポートされています. フォント設定のためのユーザ インタフェースはまだ動きませんが、prefs.jsを直接編集して非アスキーの フォント名、およびそのフォントサイズを設定すれば利用可能です. prefs.jsにもし非アスキー文字が含む場合、UTF-8でエンコーディングされている必要があるので くれぐれも注意してください. UTF-8 を扱えるエディタを見つけるのは大変かも知れません. 日本語フォント名 は日本語版Windows95/98/NT4やMacintoshの環境で動くことが確認されています. 利用するフォント名は、Windows環境であればコントロールパネルのフォント、マックであれば、 Capsキーユーティリティ、UNIXではfont family nameで表示される例の赤ボールド体の部分と同じものを 指定してください. フォント名にはスペースが入ってても構いません.:


      • -adobe-courier-medium-o-normal-0-0-75-m-0-iso8859-1

    • 韓国語や中国語のフォント名がその言語環境のOSで使えるか試してみて我々にお手伝い ください. 以下は日本語版のWindowsでprefs.jsを直接編集してフォント設定を行う方法を 示したものです. キーポイントは prefs.js が非アスキー文字を含む場合、UTF-8でエンコードされている 必要がある、という点です.:
      • 現在利用している prefs.js を見つけ, コピーしてバックアップをとります, そして オリジナルの prefs.js を例えば jpprefs.txt.のように名前を変更します.
      • メモ帳のようなテキストエディタを使って, jpprefs.txt を以下のテンプレートを利用して フォント名とフォントサイズを直接編集します:
        • user_pref("font.name.serif.ja", "Font_name_in_JPN"); -- 以下に示す５書体が設定可能です.
        • user_pref("font.size.variable.ja", 12); -- サイズは ピクセルで表されます. これはHTMLのデフォルト表示を制御します.
        • user_pref("font.size.fixed.ja", 12); -- サイズは ピクセルで表されます. これは <pre>.. </pre> で括られたテキストや プレーンテキストのメール表示のような monospace表示を制御します.
          • 選択可能なフォントタイプ: serif, sans-serif, monospace, cursive and fantasy.
          • 選択可能な言語グループは: ja, zh-TW, ko, zh-CN, x-western, x-central-euro, x-cyrillic, el, tr, x-unicode, zh-TWです.
          • 選択可能なフォントサイズカテゴリ: variable, fixed.
          • 選択可能なフォントサイズ:ピクセルの整数.

    地域化(Localizability):
    • 概要: Mozillaのキーとなる特徴の1つは、メニューやダイアログのUIを、任意の言語へ 比較的容易に地域化します。 ガイドラインの簡単な議論のために、フレームワークの節から下を見て下さい。 商用向けのブラウザは、利用者の少ない言語へ、しばしば地域化されません。 もし、あなたの母国語の速い次世代のブラウザを見たいなら、 Mozillaは、それを現実にする 素晴らしいチャンス を提供します。
    • ボランティアの地域化プロジェクト: 現在、17のボランティアの言語プロジェクトが活動しており、更に参加するらしいです。 もし、あなたのお気に入りの言語へMozillaを地域化することに興味があるのなら、 このページ が、その全てを解説しています。 現在のプロジェクトは: Bosnia, Brazilian, Catalan, Czech, Danish, French, Georgian, German, Greek, Hawaiian, Indonesian, Japanese, Norwegian (nynorsk), Polish, Spanish (es-CO), Spanish (es-ES), Thai です。 それらのいくつかは、M13キットを既に発表しています。 MozillaのL10n(地域化)のHow-toのページ をチェックして下さい。
    • 地域化のフレームワーク:
      • 私達は、地域化のガイドラインを公表しました。 その他の言語へMozillaを地域化することに興味を持った人は、 このドキュメント を注意深く読まなければなりません。
      • Mozillaを地域化する時、心に留める基本的な考え方がここにあります。 Mozilla全体を翻訳するために、 このドキュメント を読み、 そして、より早いバージョンから、翻訳同期化ツール(leveraging tool to synch up) を使って、管理された方法でそうしなければなりません。 ちょっと2、3の句を翻訳する時、次のステップに従って下さい。
        • .xulファイルを直接地域化しないで下さい。 全ての地域化は、対応した.dtdファイルや、.propertyを 置き換える必要があります。
        • 地域化した文字列は、.xulファイルから抜かれ、 .dtd、.propertyファイルの中へ、現在、移されました。 それらのファイルは、 ../chrome/component_name/locale/en-US ディレクトリの下に見付かります。 それらは、 ../chrome/component_name/content/defaultdirectory の下に置かれた対応する.xulファイルの名前に一致します。
        • 翻訳者はこれについて気にする必要はありませんが、もし、.xulファイルを作成するなら、 地域化された文字列を、対応する.dtdや.property ファイルへ抜き出す必要があります。 (固定的な.xulファイルによってよりも、scriptによってMozilla UIを作っているならば、 それをプロパティファイルへ抜き出す必要があります。) 地域化のために.XULのentityをDTDファイルへ抜き出すためには、 このドキュメント を読んで下さい。
        • Mozillaは、デフォルトの文字セットが、全ての.dtdファイルはUTF-8であり、 全ての.property ファイルはescaped Unicodeであると仮定しています。 もし、UIの文字列をnon-ASCIIのdataを使ったあなたの言語に変えるのならば、 それらを適切なUnicode encoding formatを使って。 (メニューを日本語へ変更できます。例えば、 上で提案された方法を使っている..chrome/navigator/locale/en-US/navigator.dtd ファイルで、DTDファイルをUTF-8へ変換して下さい。) メニューの項目を一般に様々な言語にすることは、あなたのシステムではサポートしていません。 例えば、ちょうど今は、US Windowsの日本語のメニューは可能ではありません。
        • 全てのnon-ASCII .dtdファイルは、UTF-8 encodingで、 .propertyファイルは、escaped Unicodeでなければなりません -- それらはリソースファイルのためMozillaのdefault encodingです。 それらのencodingのリソースファイルは、明示的にマークを付ける必要はありませんし、 charsetが無ければ、Mozillaは、それらはUTF-8または escaped Unicodeであると仮定するでしょう。

          • Mozillaのnsconv変換ユーティリティ は、"jonas.utterstrom@vittran.norrnod.se"によって機能が向上しました。 -- それはcharsetの別名を現在サポートしています。: オリジナルのnsconvは、 Frank Tang のサービスによるものです。 nsconvは、あなたのmozillaを実行するdirectoryと同じところへインストールします。 Mozillaの認識できるencodingのどれか、または、使われているこのユーティリティの ソースの中でチェックされているそれの別名を使うことができます。 このユーティリティを使うための基本的なコマンドラインはここです。:
            • schemaの使い方 --次の1つ:
              • nsconv -f source_charsetname -t target_charsetname source_filename new_filename
              • nsconv -f source_charsetname -t target_charsetname source_filename > new_filename
              • nsconv -f source_charsetname -t target_charsetname < source_filename > new_filename
            • source_charsetname と target_charsetnameのために、 文字セットのメニューの括弧の中に見られる 文字セットの名前を使えます。 例えば、iso-8859-1, Shift_JIS, Big5, EUC-KRなど。 target encodingにはUTF-8とx-u-escapedを使って下さい。

    Latin 1 と日本語のメッセージ作成の際の注意:
      • Latin 1 メールの作成:
        • アクセント付き文字については、コピーを実行できるアプリケーションであれば、 どこからでも(例:メッセンジャー(Messenger)のメールウィンドウ、ブラウザウィンドウ)、 ヘッダや本文へコピー/貼り付けを行うことが可能です。また、非ASCII文字をコミュニケー ター(Communicator) 4.xからMozillaへコピーすることもできます。
        • キーボードを使って、題名(Subject)などのヘッダ項目にアクセント付き文字を入力する ことも可能です。英語キーボードを使って Latin 1 の上位ビット文字を入力するには、 ALTGr + 0 + テンキー を使う方法が利用できます。(例:右ALTキー + 0232)。ただし、バ グのため、 ALT+テンキーを使うのは面倒です。詳しくは、エディタの項目を参照してく ださい。
        • メッセージを送信する前に、View | Character Setを使って文字集合の名前を忘 れずに選択してください。
        • 基本的なMIMEへの準拠: ヘッダのQエンコーディング、本文のアクセント付き文字に対する QPエンコーディングによって行なっています。
      • 日本語メールの作成:
        • 本文では基本的な日本語入力が可能です。また、ヘッダや本文への日本語の入力/コピー も、Mozilla内部からであれば可能です。さらに、コミュニケーター(Communicator)4.xか らMozillaへ日本語データをコピーすることもできます。
        • メールはISO-2022-JPで送信されます。ヘッダはBエンコーディングされます。(文字集合 切り替えには、 JISX0208-1990/83 のエスケープシーケンスを用います。)
        • メッセージを送信する前に、View | Character Setを使って日本語 (ISO-2022-JP)を忘れずに選択してください。
      • 他の文字集合によるメールの送信は可能です。 それらの新しい文字集合(例:中央ヨーロッパ文字、キリル文字、ギリシャ文字、UTF-8、タ イ文字、その他)について、試してください！
        • メール本文の入力においては、キーボードの種類を判別してフォントを適宜変更すること が可能ですが、いくつかの国際化されたキーボードに関しては、対応付けにバグがあるか もしれません。もし、あなたが使っている文字集合/言語でそのようなバグを発見したら、 こちらに送ってください。
